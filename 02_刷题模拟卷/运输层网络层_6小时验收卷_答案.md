### 2. 停止等待与可靠传输协议（共8分）

（1）停止等待协议中，发送方发完一个分组后必须等待什么？（3分）

**答案**：发送方每发完一个分组后必须停下来等待接收方的确认（ACK），收到ACK后才能发送下一个分组。

**主要缺点**：信道利用率低。尤其在RTT较大时，大部分时间都在等待确认，信道几乎空转。

**评分点**：提到"等待ACK/确认"得1分；提到"信道利用率低"得1分；解释RTT影响得1分。

（2）GBN协议（3分）

**答案**：
- 接收方只按序接收，期望收到帧0。收到乱序的帧1,2,3时，接收方**丢弃**这些帧并对最后一个按序收到的帧（帧6）重发ACK。
- 发送方的计时器（为最早未确认分组帧0维护）超时后，从帧0开始**回退重传**帧0,1,2,3。

**评分点**：提到"丢弃乱序帧"得1分；提到"重发对帧6的ACK"得1分；提到"重传帧0,1,2,3"得1分。

**常见错误**："接收方缓存帧1,2,3然后发ACK3"——这是SR的行为，不是GBN。

（3）GBN计时器机制（2分）

**答案**：**B**

A错：GBN通常只维护一个计时器，不是每帧一个（那是SR）。
C不准确：不是"整个窗口统一的发送计时器"，而是为最早未确认分组维护。
D错：GBN依赖计时器触发超时重传。

**评分点**：选B得2分。

### 3. TCP seq/ack计算（共10分）

（1）B正确接收后回复的确认号ack是多少？（3分）

**步骤**：
- B收到的最后一个字节序号 = 500 + 800 - 1 = 1299
- ack = 期望收到的下一个字节序号 = 1299 + 1 = **1300**

**答案**：ack = 1300
**评分点**：公式正确（seq+数据长度=ack）得2分；数值正确得1分。

（2）A下一个报文段seq是多少？（2分）

**步骤**：下一个报文段第一个数据字节 = 500 + 800 = **1300**

**答案**：seq = 1300
**评分点**：正确得2分。

（3）不携带数据的SYN报文段（5分）

**关键规则**：SYN标志消耗1个序号，即使不携带数据。

**步骤**：
- SYN消耗1个序号 → B收到的"逻辑字节"就是序号1000本身
- B回复的ack = 1000 + 1 = **1001**
- A下一个报文段的seq = 1000 + 1 = **1001**

**答案**：ack = 1001，下一个seq = 1001
**评分点**：明确"SYN消耗1个序号"得2分；ack正确得1.5分；seq正确得1.5分。

**常见错误**：
- "SYN不携带数据所以ack=1000"——错，SYN消耗序号。
- "下一个seq=1000"——错，SYN已消耗序号1000。

### 4. RTT与RTO计算（共6分）

已知：旧RTTs=60ms，旧RTTd=20ms，新RTT样本=100ms，α=1/8，β=1/4。

（1）新的RTTs（2分）

**步骤**：
```
新RTTs = (1 - 1/8) × 60 + (1/8) × 100
       = 0.875 × 60 + 0.125 × 100
       = 52.5 + 12.5
       = 65 ms
```

**答案**：65 ms
**评分点**：公式正确得1分；数值正确得1分。

**常见错误**：α=1/8但写成乘60而不是乘7/8×60。

（2）新的RTTd（2分）

**步骤**：
```
新RTTd = (1 - 1/4) × 20 + (1/4) × |60 - 100|
       = 0.75 × 20 + 0.25 × 40
       = 15 + 10
       = 25 ms
```

**答案**：25 ms
**评分点**：使用|RTTs-样本|（不是|样本-RTTs|，但绝对值一样）得1分；数值正确得1分。

**常见错误**：忘记绝对值符号，用(60-100)=-40代入导致结果错误。

（3）新的RTO（2分）

**步骤**：
```
RTO = RTTs + 4 × RTTd
    = 65 + 4 × 25
    = 65 + 100
    = 165 ms
```

**答案**：165 ms
**评分点**：公式正确得1分；数值正确得1分。

**验证**：RTO=165ms > RTTs=65ms ✓，RTO远大于RTTs合理（考虑了波动）。

### 5. TCP拥塞控制（共10分）

已知：cwnd=24 MSS，ssthresh=16 MSS。

（1）当前处于什么阶段？（3分）

**判断**：cwnd(24) > ssthresh(16)，因此处于**拥塞避免**阶段（线性增长）。

**评分点**：正确回答"拥塞避免"得2分；给出判断依据得1分。

（2）超时重传（3分）

**步骤**：
```
ssthresh = cwnd / 2 = 24 / 2 = 12 MSS
cwnd = 1 MSS
重新进入慢开始
```

**答案**：新ssthresh=12 MSS，新cwnd=1 MSS，重新进入慢开始
**评分点**：ssthresh正确得1分；cwnd=1正确得1分；提到"慢开始"得1分。

（3）3个重复ACK（按课程简化模型）（4分）

**步骤**：
```
ssthresh = cwnd / 2 = 24 / 2 = 12 MSS
cwnd = ssthresh = 12 MSS
进入拥塞避免阶段
```

**答案**：新ssthresh=12 MSS，新cwnd=12 MSS，进入拥塞避免

> **模型说明**：以上按课程常用简化模型。TCP Reno标准(RFC 5681)：进入快恢复时 cwnd = ssthresh + 3×SMSS，快恢复期间每收到一个重复ACK cwnd += SMSS，收到新数据ACK后将cwnd降到ssthresh并退出快恢复。考试以任课教师模型为准。

**评分点**：ssthresh正确得1分；cwnd正确得1分；提到"拥塞避免"得1分；理解快重传+快恢复机制得1分。

**常见错误**：将快恢复的cwnd写成1（那是超时重传的处理方式）。

### 6. TCP连接管理（共8分）

（1）为什么需要三次握手？（4分）

**答案**：
三次握手可以防止已失效的旧连接请求报文段（historical duplicate SYN）突然到达服务器，导致服务器错误建立连接。

具体过程：如果只有两次握手，一个在网络中延迟很久的旧SYN报文段可能突然到达服务器。服务器收到后回复SYN+ACK并分配资源，认为连接已建立。但客户端早已不需要该连接，不会回复ACK。服务器资源被白白占用。

第三次握手让客户端确认自己确实需要建立连接，服务器收到客户端ACK后才分配资源，避免了旧请求的干扰。

**评分点**：提到"防止旧/失效连接请求"得2分；解释两次握手的问题得1分；解释第三次握手如何解决得1分。

（2）TIME-WAIT（4分）

**答案**：
主动关闭方发送最后一个ACK后进入**TIME-WAIT**状态，等待**2MSL**（两倍最长报文段寿命）。

两个作用：
1. **保证最后一个ACK可靠到达对方**：如果最后一个ACK丢失，被动关闭方会超时重传FIN，主动关闭方仍处于TIME-WAIT状态可以再次回复ACK。
2. **让旧连接报文段从网络中消失**：等待2MSL使本连接中滞留在网络中的旧报文段全部失效，避免它们干扰后续使用相同四元组（源IP、源端口、目的IP、目的端口）的新连接。

**评分点**：提到TIME-WAIT得1分；提到2MSL得1分；两个作用各1分。

**步骤**：主机位 = 32 - 18 = 14，可用 = 2^14 - 2 = 16384 - 2 = 16382

**答案**：16382个（计算式：2^14 - 2 = 16382）

**评分点**：每题2分。掩码正确1分+格式正确1分。网络/广播地址各2分（二进制转换过程可写在草稿纸上）。可用主机数公式1分+数值1分。

### 10. RIP路由更新（共8分）

R3当前路由表：
| 目的 | 距离 | 下一跳 |
|---|---|---|
| N1 | 2 | R1 |
| N2 | 5 | R2 |
| N3 | 1 | 直连 |

R1更新：
| 目的 | R1距离 |
|---|---|
| N1 | 3 |
| N2 | 3 |
| N4 | 1 |

**更新过程**（距离向量算法）：

**N1**：R1是当前下一跳，R1报告N1距离=3。
→ RIP规则：同一下一跳的更新，无论距离变大还是变小，都更新。
→ R3经R1到N1 = 3 + 1 = 4，更新为距离4，下一跳R1。

**N2**：当前经R2距离=5。R1报告N2距离=3。
→ R3经R1到N2 = 3 + 1 = 4 < 当前5。
→ 更新为距离4，下一跳改为R1。

**N3**：R1的更新中没有N3。
→ 保持不变：距离1，直连。

**N4**：新网络。
→ R3经R1到N4 = 1 + 1 = 2，下一跳R1。

**答案**：

| 目的网络 | 距离 | 下一跳 | 更新理由 |
|---|---|---|---|
| N1 | 4 | R1 | 同一下一跳R1，更新距离为3+1=4 |
| N2 | 4 | R1 | 经R1距离4 < 原经R2距离5 |
| N3 | 1 | 直连 | R1更新中无N3，保持不变 |
| N4 | 2 | R1 | 新网络，经R1距离1+1=2 |

**评分点**：每条路由2分（距离正确1分+下一跳正确1分）。写出更新理由可帮助判分。

**常见错误**：N1认为距离不变（忘记同一下一跳更新规则）、忘记加1（"R1距离3→R3经R1=3"）。

### 11. IP数据报分片（共10分）

已知：总长度5000字节，首部20字节，MTU=1300字节，DF=0。

（1）分几片？（2分）

**步骤**：
- 原始数据 = 5000 - 20 = 4980字节
- 每片最大数据 = floor((1300 - 20) / 8) × 8 = floor(1280/8) × 8 = 160 × 8 = 1280字节
- 第1片：1280字节
- 第2片：1280字节
- 第3片：1280字节
- 第4片：4980 - 1280×3 = 4980 - 3840 = 1140字节
- 共4片

**答案**：4片

**评分点**：正确计算出每片最大数据长度1280得1分；得出4片得1分。

（2）填表（8分）

**步骤**：
- 片1：数据1280B，MF=1（后面还有），偏移=0/8=**0**
- 片2：数据1280B，MF=1，偏移=1280/8=**160**
- 片3：数据1280B，MF=1，偏移=2560/8=**320**
- 片4：数据1140B，MF=**0**（最后一片），偏移=3840/8=**480**

**答案**：

| 分片 | 数据长度(字节) | MF | 片偏移 |
|---|---|---|---|
| 第1片 | 1280 | 1 | 0 |
| 第2片 | 1280 | 1 | 160 |
| 第3片 | 1280 | 1 | 320 |
| 第4片 | 1140 | 0 | 480 |

**验证**：数据总长 = 1280×3 + 1140 = 4980 = 5000-20 ✓

**评分点**：每片2分（数据长度0.5分+MF 0.5分+片偏移1分）。

**常见错误**：
- 片偏移忘记除以8（写成1280,2560,3840）

**单位提示**：片偏移字段值为偏移量/8，无量纲（或理解为"以8字节为单位"）。

### 12. ARP与NAT简答（共8分）

（1）ARP跨网查询（4分）

**答案**：
目的MAC地址是**默认网关（192.168.1.1）的MAC地址**，不是服务器S(8.8.8.8)的MAC地址。

原因：ARP协议只在同一局域网内有效，不能跨网络工作。主机A通过检查目的IP 8.8.8.8发现自己与S不在同一网络（A在192.168.1.0/24，S在公网），因此A将数据帧发给默认网关192.168.1.1。帧中的目的MAC地址是网关的MAC地址，目的IP地址仍为S的IP地址8.8.8.8。网关收到后重新封装并逐跳转发到S。

**评分点**：答出"网关MAC"得2分；解释"ARP不跨网络"得1分；解释"MAC每跳变化，IP不变"得1分。

（2）NAT主要作用（4分）

**答案**：
1. **使内网主机访问Internet**：将内部网络的私有IP地址（如192.168.x.x）和端口转换为合法的公网IP地址和端口，使使用私有地址的内网主机能够访问Internet。
2. **缓解IPv4地址短缺**：多个内网主机可以共用一个（或少数几个）公网IP地址访问互联网，大大减少对公网IPv4地址的需求。

（附加）对外隐藏内部网络拓扑结构，提供一定的安全性。

**评分点**：每个作用2分。写出两个即可。
