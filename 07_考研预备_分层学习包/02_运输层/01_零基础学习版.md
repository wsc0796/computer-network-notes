## 0. 先建立整章地图

运输层位于：

```text
应用层
运输层
网络层
数据链路层
物理层
```

你可以先这样理解：

| 层次 | 像什么 | 解决什么 |
|---|---|---|
| 网络层 IP | 快递车把包裹送到某栋楼 | 主机到主机 |
| 运输层 TCP/UDP | 楼里的前台把包裹交给具体房间/人 | 进程到进程 |
| 应用层 HTTP/DNS 等 | 真正使用包裹的人 | 具体应用服务 |

一句话：

> IP 负责找到哪台主机，端口负责找到这台主机里的哪个应用进程。

本章主线：

1. 运输层为什么需要端口。
2. UDP 怎么提供简单快速的传输。
3. TCP 怎么提供可靠、有序、面向连接的传输。
4. TCP 如何处理可靠传输、流量控制、拥塞控制、连接建立和释放。

## 1. 运输层到底解决什么问题

### 1.1 网络层只能送到主机

假设你打开浏览器，同时还开着微信、QQ、网易云。

外面发来的数据包到达你的电脑后，操作系统要知道：

```text
这个数据应该交给浏览器？
还是交给微信？
还是交给别的软件？
```

仅靠 IP 地址不够，因为 IP 只能定位到“这台电脑”。

所以运输层引入了端口号。

```text
IP 地址：定位主机
端口号：定位主机中的应用进程
```

### 1.2 运输层的最低服务

运输层最基本要做两件事：

1. 进程到进程的数据交付。
2. 差错检测。

差错检测的意思是：检查数据在传输中有没有发生比特错误。

注意：

| 层次 | 检查什么 |
|---|---|
| 网络层 IP | 主要检查 IP 首部 |
| 运输层 UDP/TCP | 检查运输层首部和数据 |
| 数据链路层 | 检查一段链路上传输的帧 |

### 1.3 运输层的增强服务

UDP 只提供很少服务，TCP 提供很多增强服务。

| 功能 | UDP | TCP |
|---|---|---|
| 可靠传输 | 不保证 | 保证 |
| 按序到达 | 不保证 | 保证 |
| 流量控制 | 没有 | 有 |
| 拥塞控制 | 没有 | 有 |
| 连接管理 | 没有 | 有 |

你可以先记：

```text
UDP：简单、快、不保证
TCP：复杂、稳、保证可靠
```

## 2. 端口和套接字

### 2.1 软件端口和硬件端口

课程里容易出现“软件端口”和“硬件端口”的区别。

| 类型 | 含义 |
|---|---|
| 硬件端口 | 设备上真实的接口，比如网线接口、USB 接口 |
| 软件端口 | 操作系统里的逻辑编号，用来标识应用进程 |

运输层说的端口，一般都是软件端口。

### 2.2 端口号范围

端口号是 16 位，所以范围是：

```text
0 - 65535
```

分三类：

| 类型 | 范围 | 用途 |
|---|---:|---|
| 熟知端口/系统端口 | 0-1023 | 常见服务器程序使用 |
| 登记端口 | 1024-49151 | 应用程序登记使用 |
| 短暂端口/动态端口 | 49152-65535 | 客户端临时使用 |

常见例子：

| 协议 | 端口 |
|---|---:|
| HTTP | 80 |
| HTTPS | 443 |
| DNS | 53 |
| FTP | 21 |
| SMTP | 25 |

### 2.3 为什么客户端也需要端口

很多零基础同学会问：

> 服务器有端口我能理解，客户端为什么也要有端口？

因为服务器回数据时，也要知道回给客户端里的哪个进程。

比如你的电脑同时打开两个浏览器标签页访问同一网站，系统要区分这些请求，就会给不同连接分配不同的临时端口。

### 2.4 UDP 套接字：二元组

UDP 套接字用二元组标识：

```text
<目的 IP 地址, 目的端口号>
```

意思是：只要数据到达同一个目的 IP 和目的端口，就交给同一个 UDP 套接字。

UDP 不为每个对方单独建立连接。

### 2.5 TCP 套接字：四元组

TCP 连接用四元组标识：

```text
<源 IP 地址, 源端口号, 目的 IP 地址, 目的端口号>
```

为什么 TCP 要四元组？

因为一个服务器的同一个端口可以同时服务很多客户端。

例如很多人都访问：

```text
服务器 IP:443
```

服务器必须通过“谁发来的”来区分不同连接，所以要看源 IP 和源端口。

## 3. UDP：简单快速的运输层协议

### 3.1 UDP 先用人话理解

UDP 像发普通明信片：

1. 写上地址就发。
2. 不提前打电话确认对方在不在。
3. 不保证一定到。
4. 不保证先发的先到。
5. 不负责丢了之后重发。

所以 UDP 的特点是：

```text
无连接、尽最大努力、面向报文、首部小、速度快
```

### 3.2 UDP 建立在 IP 之上

IP 本身是尽力而为服务：

1. 不保证交付。
2. 不保证按序。
3. 不保证不重复。
4. 不保证时延。

UDP 在 IP 之上只增加了：

1. 复用和分用。
2. 差错检测。

UDP 没有增加可靠传输、流量控制、拥塞控制。

### 3.3 UDP 面向报文

“面向报文”的意思是：

> 应用层交给 UDP 一个完整报文，UDP 就尽量保持这个报文边界，直接加首部后交给 IP。

UDP 不会像 TCP 那样把数据看成连续字节流。

对比：

| 协议 | 看待数据的方式 |
|---|---|
| UDP | 一个一个完整报文 |
| TCP | 连续不断的字节流 |

### 3.4 UDP 为什么还需要

UDP 看起来“不可靠”，但它有自己的价值。

适合 UDP 的场景：

1. 实时性强，比如语音、视频、直播、游戏。
2. 请求响应很短，比如 DNS。
3. 应用层自己能处理可靠性。
4. 不想花时间建立连接。

一句话：

> UDP 牺牲一部分可靠性，换取简单和低时延。

### 3.5 UDP 首部

UDP 首部固定 8 字节，包含 4 个字段：

| 字段   |   长度 | 作用             |
| ---- | ---: | -------------- |
| 源端口  | 2 字节 | 发送方应用进程端口，可选   |
| 目的端口 | 2 字节 | 接收方应用进程端口，必须   |
| 长度   | 2 字节 | UDP 首部 + 数据总长度 |
| 校验和  | 2 字节 | 检查首部和数据是否出错    |

考试常问：

> UDP 首部长度是多少？

答案：

```text
8 字节
```

## 4. TCP：可靠的运输层协议

### 4.1 TCP 先用人话理解

TCP 像两个人正式打电话：

1. 先建立连接：你在吗？我在。好，我们开始。
2. 通信过程中有编号和确认：我发第 1 段，你确认收到。
3. 丢了会重传。
4. 太快了会慢一点。
5. 网络堵了也会慢一点。
6. 结束时正式释放连接。

### 4.2 TCP 的主要特点

| 特点 | 含义 |
|---|---|
| 面向连接 | 通信前先建立连接 |
| 点对点 | 每条 TCP 连接只有两个端点 |
| 可靠交付 | 保证无差错、不丢失、不重复、按序到达 |
| 全双工通信 | 双方可以同时发送和接收 |
| 面向字节流 | 把应用数据看成连续字节序列 |

### 4.3 面向字节流怎么理解

应用层可能这样交数据：

```text
第一次交 5 字节
第二次交 8 字节
第三次交 2 字节
```

TCP 不关心这些边界，而是看成：

```text
一条连续的字节流
```

TCP 发送时可能重新分段。

所以：

> TCP 保证字节顺序，不保证应用层每次写入的数据边界。

### 4.4 TCP 报文段

TCP 传输的数据单位叫：

```text
TCP 报文段 segment
```

一个 TCP 报文段包括：

```text
TCP 首部 + TCP 数据部分
```

TCP 首部最小长度：

```text
20 字节
```

有选项时会更长。

## 5. TCP 首部字段零基础解释

TCP 首部字段很多，先抓重点。

### 5.1 源端口和目的端口

作用：找到发送方和接收方应用进程。

```text
源端口：谁发来的
目的端口：发给谁
```

### 5.2 序号 seq

TCP 是面向字节流的，所以序号是给字节编号。

如果 `seq = 100`，并且这个报文段携带 20 字节数据，那么它携带的是：

```text
100 - 119 号字节
```

下一个报文段通常从：

```text
seq = 120
```

开始。

### 5.3 确认号 ack

确认号表示：

> 我期望收到的下一个字节序号。

如果 `ack = 120`，意思是：

```text
119 号及之前的字节我都收到了，下一步请发 120 号字节。
```

这是累计确认。

### 5.4 数据偏移

数据偏移表示 TCP 数据部分从哪里开始。

为什么需要它？

因为 TCP 首部可能有选项，首部长度不固定，所以要告诉接收方：

```text
前面这些是 TCP 首部，从这里开始才是真正数据。
```

### 5.5 窗口 window

窗口字段用于流量控制。

接收方告诉发送方：

```text
我现在还能接收多少字节。
```

这个值叫接收窗口 `rwnd`。

### 5.6 TCP 标志位

常见标志位：

| 标志位 | 名称 | 人话解释 |
|---|---|---|
| SYN | 同步 | 请求建立连接 |
| ACK | 确认 | 确认号字段有效 |
| FIN | 结束 | 我这边没有数据要发了 |
| RST | 复位 | 连接出错，强制释放 |
| PSH | 推送 | 希望尽快交给应用层 |
| URG | 紧急 | 有紧急数据 |

考试最常用的是：

```text
SYN、ACK、FIN
```

它们分别对应：

```text
建立连接、确认、释放连接
```

### 5.7 TCP 选项

课程里提到几个常见选项。

#### MSS

MSS 是 Maximum Segment Size，最大报文段长度。

它表示：

> TCP 报文段中数据部分的最大长度。

注意：MSS 不包括 TCP 首部和 IP 首部。

MSS 太小：首部开销比例大，效率低。

MSS 太大：可能导致 IP 分片，分片丢一个就影响整个数据报，效率也差。

#### 窗口扩大

TCP 首部窗口字段只有 16 位，最大只能表示 65535。

高速网络中这个窗口可能不够大，所以使用窗口扩大选项。

实际窗口：

```text
实际窗口 = window 字段值 * 2^shift
```

#### 时间戳

时间戳主要用于：

1. 计算 RTT。
2. 防止序号绕回造成旧数据被误认为新数据。

## 6. 可靠传输：先从停止等待协议理解

TCP 的可靠传输比较复杂，课程先讲停止等待协议，是为了让你理解可靠传输的基本思想。

### 6.1 停止等待协议是什么

停止等待就是：

> 发送方每发一个分组，就停下来等确认；收到确认后，再发下一个。

流程：

```text
A 发送 M1 给 B
B 收到 M1，返回 ACK
A 收到 ACK，再发送 M2
```

优点：简单。

缺点：效率低，因为大部分时间可能在等待。

### 6.2 正常情况

```text
A -> B：发送 M1
B -> A：确认 M1
A -> B：发送 M2
B -> A：确认 M2
```

发送方每次都等确认，不会连续发送很多分组。

### 6.3 数据分组丢失

如果 A 发出的 M1 丢失：

```text
A -> B：M1 丢失
B：收不到 M1，所以不会确认
A：等待超时
A：重传 M1
```

关键机制：

```text
超时重传
```

### 6.4 确认丢失

如果 B 收到了 M1，也发了 ACK，但 ACK 丢失：

```text
A -> B：M1 到达
B -> A：ACK 丢失
A：等待超时
A：重传 M1
B：发现 M1 重复，丢弃重复数据，再发 ACK
```

这里就需要分组编号。

因为 B 要判断：

```text
这个 M1 是新数据，还是之前收到过的重复数据？
```

### 6.5 确认迟到

如果 ACK 没丢，只是来晚了：

```text
A -> B：M1 到达
B -> A：ACK 很晚才到
A：先超时，于是重传 M1
B：收到重复 M1，丢弃，再发 ACK
A：后来可能收到迟到的 ACK
```

发送方需要能识别迟到的确认，避免混乱。

### 6.6 停止等待协议的要点

| 问题 | 解决方法 |
|---|---|
| 数据丢失 | 超时重传 |
| 确认丢失 | 超时重传 + 编号去重 |
| 确认迟到 | 编号判断 |
| 重复数据 | 接收方丢弃重复分组 |

### 6.7 ARQ

ARQ 是 Automatic Repeat reQuest，自动重传请求。

意思是：

> 发送方在超时后自动重传，不需要接收方单独发“请重传”命令。

### 6.8 信道利用率为什么低

停止等待协议的效率低，因为发送方发送完一个分组后，大量时间都在等 ACK。

如果 RTT 很长，数据真正发送的时间很短，等待时间很长，信道利用率就低。

改进方法：

```text
流水线传输
```

也就是：不要等一个确认才发下一个，而是在窗口允许范围内连续发送多个分组。

这就引出滑动窗口。

## 7. TCP 可靠传输：滑动窗口

### 7.1 为什么需要滑动窗口

停止等待太慢。

所以 TCP 使用滑动窗口，让发送方可以连续发送多个字节范围内的数据。

可以这样理解：

```text
停止等待：发一个，等一个
滑动窗口：一批一批发，确认到了窗口再往前滑
```

### 7.2 TCP 按字节编号

TCP 不是按报文段编号，而是按字节编号。

例如发送 1000 字节：

```text
第 1 个字节编号 1
第 2 个字节编号 2
...
第 1000 个字节编号 1000
```

这就是“面向字节流”的体现。

### 7.3 发送窗口

发送窗口表示：

> 发送方当前被允许发送的字节范围。

窗口内的数据分几类：

| 区域 | 状态 |
|---|---|
| 已发送且已确认 | 可以删除 |
| 已发送但未确认 | 不能删除，可能要重传 |
| 允许发送但尚未发送 | 可以发 |
| 窗口外 | 暂时不能发 |

### 7.4 接收窗口

接收窗口表示：

> 接收方当前愿意接收的字节范围。

接收方按序交付给应用层。

如果中间缺了一段，后面到了也可能要先缓存，不能直接交付。

### 7.5 累计确认

TCP 常用累计确认。

```text
ACK = x
```

表示：

```text
x 之前的所有字节都收到了，下一个想要 x。
```

例子：

```text
ACK = 1001
```

表示：

```text
1 - 1000 都收到了，下一步要 1001。
```

### 7.6 选择确认 SACK

普通累计确认有一个问题：只能告诉发送方“从哪里开始缺”，不能告诉它“后面哪些已经收到了”。

例如接收方收到：

```text
1-1000
1501-3000
3501-4500
```

缺：

```text
1001-1500
3001-3500
```

普通 ACK 可能只能反复说：

```text
ACK = 1001
```

发送方不知道后面哪些已经收到，可能多重传。

SACK 可以告诉发送方：

```text
这些不连续的数据块我已经收到了，你只重传缺的部分。
```

## 8. 超时重传时间 RTO

### 8.1 RTO 为什么不能乱设

RTO 是超时重传时间。

| RTO | 后果 |
|---|---|
| 太短 | 明明没丢，只是慢，也会误重传 |
| 太长 | 真丢了也等很久，效率低 |

所以 TCP 要根据 RTT 动态计算 RTO。

### 8.2 RTT

RTT 是往返时间：

```text
发送一个报文段 -> 收到对应确认
```

这段时间就是 RTT。

### 8.3 加权平均 RTT

TCP 不只看最新一次 RTT，因为一次测量可能偶然偏大或偏小。

所以用加权平均：

```text
新的 RTTs = (1 - alpha) * 旧 RTTs + alpha * 新 RTT 样本
```

常见：

```text
alpha = 1/8
```

### 8.4 RTT 偏差和 RTO

网络波动越大，RTO 应该越保守。

常见公式：

```text
RTO = RTTs + 4 * RTTd
```

其中 `RTTd` 表示 RTT 偏差。

### 8.5 Karn 算法

如果一个报文段被重传过，后来收到了 ACK，问题来了：

```text
这个 ACK 是确认原来的报文段？
还是确认重传的报文段？
```

无法判断。

所以 Karn 算法规定：

> 被重传过的报文段，不用它的 ACK 来计算 RTT 样本。

修正 Karn 算法：

> 每次重传，就把 RTO 加倍。

常见：

```text
新的 RTO = 2 * 旧 RTO
```

## 9. TCP 流量控制

### 9.1 流量控制解决什么

流量控制解决的是：

> 发送方不要发太快，把接收方缓存撑爆。

注意：它看的是接收方能力。

### 9.2 接收窗口 rwnd

接收方通过 TCP 首部窗口字段告诉发送方：

```text
我还能接收多少字节。
```

这个窗口叫：

```text
rwnd
```

发送方必须控制发送量，不超过接收方通告的窗口。

### 9.3 零窗口问题

如果接收方缓存满了，会告诉发送方：

```text
rwnd = 0
```

发送方停止发送。

但如果之后接收方窗口变大，通知报文丢了，发送方就会一直以为窗口为 0。

这就是零窗口死锁风险。

### 9.4 持续计时器

解决方法：持续计时器。

当发送方收到零窗口通知后，启动持续计时器。

计时器到期后，发送方发送零窗口探测报文，问：

```text
你现在窗口还是 0 吗？
```

接收方回复当前窗口大小。

### 9.5 糊涂窗口综合征

糊涂窗口综合征指：

> TCP 一次只发送很少几个字节，导致首部开销很大，效率很低。

例如只发 1 字节数据，却要加上 20 字节 TCP 首部和 20 字节 IP 首部，非常浪费。

### 9.6 Nagle 算法

Nagle 算法解决发送方频繁发送小报文段的问题。

核心思想：

> 如果前面的小报文段还没确认，就先攒一攒，别急着继续发小包。

## 10. TCP 拥塞控制

### 10.1 拥塞控制解决什么

拥塞控制解决的是：

> 不要往网络里塞太多数据，避免路由器和链路过载。

注意：它看的是网络整体承受能力。

### 10.2 流量控制 vs 拥塞控制

| 对比 | 流量控制 | 拥塞控制 |
|---|---|---|
| 看谁 | 接收方 | 网络 |
| 目的 | 不撑爆接收缓存 | 不压垮网络 |
| 关键窗口 | rwnd | cwnd |
| 控制依据 | 接收方通告 | 丢包、RTT、重复 ACK 等 |

一句话：

```text
流量控制：怕接收方吃不下
拥塞控制：怕网络路上堵车
```

### 10.3 拥塞窗口 cwnd

发送方维护拥塞窗口：

```text
cwnd
```

实际发送窗口上限：

```text
发送窗口 = Min(rwnd, cwnd)
```

也就是说：

1. 接收方窗口小，就听接收方的。
2. 网络拥塞窗口小，就听网络的。

发送速率近似：

```text
rate = cwnd / RTT
```

### 10.4 慢开始

慢开始不是“增长慢”，而是：

> 一开始窗口小，然后指数增长。

典型变化：

```text
1, 2, 4, 8, 16 ...
```

每经过一个 RTT，`cwnd` 大致翻倍。

### 10.5 拥塞避免

当 `cwnd` 增长到慢开始门限 `ssthresh` 后，改用拥塞避免。

拥塞避免是线性增长：

```text
每个 RTT 大约增加 1 个 MSS
```

对比：

| 阶段 | 增长方式 |
|---|---|
| 慢开始 | 指数增长 |
| 拥塞避免 | 线性增长 |

### 10.6 慢开始门限 ssthresh

规则：

| 条件 | 使用算法 |
|---|---|
| cwnd < ssthresh | 慢开始 |
| cwnd > ssthresh | 拥塞避免 |
| cwnd = ssthresh | 两者都可 |

### 10.7 超时后的处理

如果发生超时重传，通常认为网络拥塞较严重。

常见处理：

```text
ssthresh = cwnd / 2
cwnd = 1 MSS
重新慢开始
```

### 10.8 快重传

如果发送方连续收到 3 个重复 ACK，说明某个报文段很可能丢了。

这时不等超时，立即重传。

```text
3 个重复 ACK -> 快重传
```

### 10.9 快恢复

收到 3 个重复 ACK，说明后面的报文段还能到达，网络没有完全瘫痪。

所以通常不把 `cwnd` 降到 1，而是：

```text
ssthresh = cwnd / 2
cwnd = ssthresh
进入拥塞避免
```

### 10.10 AQM 和 RED

传统路由器尾部丢弃：

> 队列满了才丢弃后来的分组。

问题：多个 TCP 连接可能同时发现丢包，同时降速，再同时升速，形成全局同步。

AQM 主动队列管理：

> 队列还没满时，就提前随机丢弃或标记一些分组，让发送方提前降速。

RED 是一种典型 AQM 算法：

```text
Random Early Detection，随机早期检测
```

## 11. TCP 连接建立：三次握手

### 11.1 三次握手先用人话理解

三次握手像这样：

```text
客户端：你在吗？我想连接。
服务器：我在，我同意连接，你听得到我吗？
客户端：我听得到，我们开始。
```

### 11.2 三次握手过程

第一步：

```text
客户端 -> 服务器：SYN = 1, seq = x
```

含义：客户端请求建立连接，并选择自己的初始序号 x。

第二步：

```text
服务器 -> 客户端：SYN = 1, ACK = 1, seq = y, ack = x + 1
```

含义：服务器同意连接，也选择自己的初始序号 y，并确认客户端的 SYN。

第三步：

```text
客户端 -> 服务器：ACK = 1, seq = x + 1, ack = y + 1
```

含义：客户端确认服务器的 SYN。

连接建立后，双方进入：

```text
ESTABLISHED
```

### 11.3 为什么必须三次

三次握手主要为了：

1. 确认双方发送能力正常。
2. 确认双方接收能力正常。
3. 交换并确认初始序号。
4. 防止失效的旧连接请求突然到达服务器，造成错误连接。

如果只有两次握手，旧的 SYN 报文可能在网络中延迟很久后到达服务器，服务器误以为客户端要建立新连接，从而浪费资源。

考试表达：

> 三次握手可以防止已失效的连接请求报文段突然传到服务器，造成服务器错误建立连接。

## 12. TCP 连接释放：四次挥手

### 12.1 四次挥手先用人话理解

TCP 是全双工的，两个方向可以独立发送数据。

所以一方说“我不发了”，不代表另一方也不发了。

四次挥手像这样：

```text
A：我没有数据要发了。
B：知道了，但我可能还有数据要发。
B：我也发完了。
A：知道了。
```

### 12.2 四次挥手过程

第一步：

```text
A -> B：FIN = 1, seq = u
```

A 主动关闭，表示自己没有数据要发了。

第二步：

```text
B -> A：ACK = 1, ack = u + 1
```

B 确认 A 的关闭请求，但 B 可能还有数据要发。

第三步：

```text
B -> A：FIN = 1, seq = v
```

B 也发完了，请求关闭自己的发送方向。

第四步：

```text
A -> B：ACK = 1, ack = v + 1
```

A 确认 B 的关闭请求。

### 12.3 为什么是四次

因为 TCP 是全双工通信。

一方的 FIN 只表示：

```text
我这个方向不发了。
```

另一方可能还要继续发送数据，所以对方要先 ACK，等自己也发完后再发 FIN。

### 12.4 TIME-WAIT

主动关闭方发送最后一个 ACK 后，不会立即关闭，而是进入 TIME-WAIT。

等待时间通常是：

```text
2MSL
```

MSL 是最长报文段寿命。

等待 2MSL 的原因：

1. 保证最后一个 ACK 能到达对方。如果 ACK 丢了，对方会重传 FIN，主动关闭方还能再次 ACK。
2. 让旧连接中的报文段从网络中消失，避免影响新连接。

考试表达：

> TIME-WAIT 的作用是保证连接可靠释放，并防止旧连接报文段影响之后的新连接。

## 13. 最容易混的概念

### 13.1 UDP 和 TCP

| 对比 | UDP | TCP |
|---|---|---|
| 连接 | 无连接 | 面向连接 |
| 可靠性 | 不保证可靠 | 可靠传输 |
| 数据形式 | 面向报文 | 面向字节流 |
| 首部 | 8 字节 | 最小 20 字节 |
| 速度 | 快、开销小 | 较复杂、开销大 |
| 适用 | 实时、简单请求 | 文件、网页、邮件等可靠场景 |

### 13.2 序号和确认号

| 字段 | 含义 |
|---|---|
| seq | 当前报文段第一个数据字节的编号 |
| ack | 期望收到的下一个字节编号 |

记忆：

```text
seq：我发的数据从几号开始
ack：我下次想收几号
```

### 13.3 rwnd 和 cwnd

| 窗口 | 中文 | 由谁决定 | 解决什么 |
|---|---|---|---|
| rwnd | 接收窗口 | 接收方 | 接收方吃不下 |
| cwnd | 拥塞窗口 | 发送方根据网络状态 | 网络太堵 |

实际发送窗口：

```text
Min(rwnd, cwnd)
```

### 13.4 超时重传和快重传

| 机制 | 触发 |
|---|---|
| 超时重传 | 等不到确认，计时器超时 |
| 快重传 | 收到 3 个重复 ACK |

### 13.5 三次握手和四次挥手

| 过程 | 次数 | 原因 |
|---|---:|---|
| 建立连接 | 3 次 | 双方确认收发能力，交换初始序号，防止旧请求 |
| 释放连接 | 4 次 | TCP 全双工，两个方向分别关闭 |

