## 1. 概念判断题

### 必背结论

```text
运输层提供进程到进程通信。
端口号用于标识主机中的应用进程。
UDP 无连接、不可靠、面向报文，首部 8 字节。
TCP 面向连接、可靠、面向字节流，首部最小 20 字节。
TCP 有流量控制和拥塞控制，UDP 没有。
TCP 面向字节流，UDP 面向报文，不能背反。
```

### 常见陷阱

| 题干 | 判断 | 原因 |
|---|---|---|
| TCP 是面向报文的 | 错 | TCP 面向字节流 |
| UDP 提供可靠交付 | 错 | UDP 尽最大努力交付 |
| 端口号只在本地主机有意义 | 对 | 端口用于本机进程标识 |
| UDP 首部最小 20 字节 | 错 | UDP 首部固定 8 字节 |
| TCP 首部固定 20 字节 | 错 | TCP 首部最小 20 字节，有选项会变长 |

### 学习通真题（老师作业，7题）

| 题号 | 题干 | 正确答案 | 陷阱/易混点 |
|---:|---|---|---|
| 1 | 应用层进程通过什么与运输层实体交互？ | **端口** | 不是IP地址、不是Socket（Socket是编程接口，端口是协议概念） |
| 2 | 端口号的长度是多少位？ | **16位**（0~65535） | UDP源端口/目的端口各2字节=16位，要能换算 |
| 3 | 运输层不可靠传输协议是什么？ | **UDP** | UDP不保证可靠交付，但可由应用层自己实现可靠机制 |
| 4 | TCP和UDP哪个通常效率更高？为什么？ | **UDP**，无连接、首部小（8字节）、无流控和拥塞控制 | 效率高≠可靠，UDP牺牲可靠性换效率 |
| 6 | 以下哪项不是UDP的特性？ | **提供可靠服务** | UDP的特性：无连接、尽最大努力、面向报文、首部8字节、支持校验和 |
| 7 | "TCP是面向报文的，UDP是面向字节流的"对吗？ | **错** | TCP面向字节流（不保留报文边界），UDP面向报文（保留报文边界） |
| 8 | "网络层提供数据报或虚电路服务不影响运输层的运行机制，但影响服务质量"对吗？ | **对** | 机制看运输层自己（TCP/UDP协议），质量受网络层影响（丢包、时延、吞吐率） |

## 2. UDP/TCP 首部字段题

### UDP 首部

UDP 首部固定 8 字节：

| 字段 | 长度 | 作用 |
|---|---:|---|
| 源端口 | 2 字节 | 发送方应用进程端口 |
| 目的端口 | 2 字节 | 接收方应用进程端口 |
| 长度 | 2 字节 | UDP 首部 + 数据 |
| 校验和 | 2 字节 | 检错 |

### TCP 首部关键字段

| 字段 | 作用 |
|---|---|
| 源端口/目的端口 | 标识通信进程 |
| seq | 本报文段数据第一个字节编号 |
| ack | 期望收到的下一个字节编号 |
| 数据偏移 | 指出 TCP 数据从哪里开始 |
| 窗口 | 接收窗口，用于流量控制 |
| SYN | 建立连接 |
| ACK | 确认号有效 |
| FIN | 释放连接 |
| RST | 复位连接 |

### seq/ack 计算模板

题目给：

```text
seq = a
数据长度 = L
```

则下一个序号：

```text
next seq = a + L
```

如果接收方成功收到这些字节，确认号：

```text
ack = a + L
```

注意：

```text
SYN 和 FIN 虽然不携带普通数据，但各消耗 1 个序号。
纯 ACK 不消耗序号。
```

## 3. RDT 与停止等待题

### RDT 递进

| 协议 | 信道问题 | 新增机制 |
|---|---|---|
| RDT 1.0 | 信道完全可靠 | 无需额外机制 |
| RDT 2.0 | 分组可能出错 | ACK/NAK |
| RDT 2.1 | ACK/NAK 也可能出错 | 序号 |
| RDT 2.2 | 不使用 NAK | 重复 ACK 表达未收到 |
| RDT 3.0 | 分组可能丢失 | 定时器、超时重传 |

### 停止等待时序

必须能画四种情况：

```text
1. 正常：发送 M1 -> ACK -> 发送 M2
2. 数据丢失：M1 丢失 -> 超时 -> 重传 M1
3. ACK 丢失：B 收到 M1 -> ACK 丢失 -> A 超时重传 -> B 丢弃重复 M1
4. ACK 迟到：A 先超时重传，后收到迟到 ACK，需要靠序号避免混乱
```

### 信道利用率

常见思想：

```text
停止等待信道利用率低，因为发送一个分组后必须等待 ACK，RTT 越大，等待浪费越明显。
```

若题目给公式型数据，先找：

```text
发送时延
RTT
确认帧发送时延是否忽略
```

常见简化：

```text
利用率 = 发送时延 / (发送时延 + RTT)
```

## 4. GBN 与 SR 题

### GBN

关键词：

```text
Go-Back-N
接收窗口通常为 1
只按序接收
乱序帧丢弃
累计确认
超时后从丢失帧开始回退重传
```

### SR

关键词：

```text
Selective Repeat
接收方可以缓存乱序帧
逐个确认
只重传丢失帧
接收方缓存和逻辑更复杂
```

### 最大窗口

若序号位数为 n：

```text
GBN 最大发送窗口 <= 2^n - 1
SR 最大窗口 <= 2^(n-1)
```

原因：

```text
窗口过大时，新旧帧序号会混淆，接收方无法判断是新一轮数据还是旧数据重传。
```

### 对比表

| 对比 | GBN | SR |
|---|---|---|
| 接收乱序帧 | 丢弃 | 缓存 |
| 确认方式 | 累计确认 | 逐个确认 |
| 重传方式 | 从丢失处开始全部重传 | 只重传丢失帧 |
| 接收窗口 | 通常为 1 | 大于 1 |
| 实现复杂度 | 较低 | 较高 |

## 5. TCP 可靠传输题

### 必会机制

```text
以字节为单位编号
累计确认
超时重传
快重传
RTT/RTO 估计
Karn 算法
SACK 选择确认
```

### ACK 规则

```text
ACK = x 表示 x 之前的字节都已经收到，下一步期望收到 x。
```

例：

```text
ACK = 1001
表示 1-1000 都收到了，下一步要 1001。
```

### Karn 算法

```text
重传过的报文段，其 ACK 不用于更新 RTT。
因为无法判断 ACK 对应原始报文段还是重传报文段。
```

修正规则：

```text
每次超时重传，RTO 加倍。
```

### SACK

```text
SACK 通过 TCP 选项告诉发送方哪些乱序数据块已经收到。
作用是只重传缺失数据，减少不必要重传。
```

## 6. RTT/RTO 计算题

### 公式

```text
新的 RTTs = (1 - alpha) * 旧 RTTs + alpha * 新 RTT 样本
alpha = 1/8
```

```text
新的 RTTd = (1 - beta) * 旧 RTTd + beta * |RTTs - 新 RTT 样本|
beta = 1/4
```

```text
RTO = RTTs + 4 * RTTd
```

### 做题步骤

```text
1. 看题目给的是旧 RTTs、旧 RTTd 还是只给 RTT 样本。
2. 先算新的 RTTs。
3. 再算新的 RTTd。
4. 最后算 RTO。
5. 如果报文段被重传过，注意 Karn 算法，不用该 ACK 更新 RTT。
```

## 7. 流量控制题

### 核心结论

```text
流量控制用于防止发送方发送太快，使接收方来不及接收。
```

关键词：

```text
rwnd
接收窗口
窗口字段
零窗口
持续计时器
零窗口探测
```

### 零窗口题模板

如果题目问零窗口：

```text
接收方缓存满时通告 rwnd = 0，发送方停止发送。
为防止窗口更新报文丢失导致双方永久等待，发送方启动持续计时器。
计时器到期后发送零窗口探测报文，接收方回复当前窗口大小。
```

## 8. 拥塞控制题

### 必背公式

```text
发送窗口 = min(rwnd, cwnd)
```

```text
rate = cwnd / RTT
```

### 慢开始与拥塞避免

| 阶段 | 增长方式 |
|---|---|
| 慢开始 | 指数增长 |
| 拥塞避免 | 线性增长 |

规则：

```text
cwnd < ssthresh：慢开始
cwnd > ssthresh：拥塞避免
cwnd = ssthresh：两者均可
```

### 拥塞事件处理

超时：

```text
ssthresh = cwnd / 2
cwnd = 1 MSS
重新慢开始
```

3 个重复 ACK：

```text
快重传 + 快恢复
ssthresh = cwnd / 2
cwnd = ssthresh
进入拥塞避免
```

### cwnd 曲线判断

看到曲线：

```text
cwnd 突然降到 1：通常是超时。
cwnd 降到约一半：通常是 3 个重复 ACK / 快恢复。
```

## 9. TCP 连接管理题

### 三次握手

```text
1. 客户端 -> 服务器：SYN=1, seq=x
2. 服务器 -> 客户端：SYN=1, ACK=1, seq=y, ack=x+1
3. 客户端 -> 服务器：ACK=1, seq=x+1, ack=y+1
```

作用：

```text
确认双方收发能力
交换初始序号
防止已失效连接请求突然到达服务器造成错误连接
```

### 四次挥手

```text
1. 主动关闭方 -> 被动关闭方：FIN
2. 被动关闭方 -> 主动关闭方：ACK
3. 被动关闭方 -> 主动关闭方：FIN
4. 主动关闭方 -> 被动关闭方：ACK
```

为什么四次：

```text
TCP 是全双工通信，一方不发送数据，不代表另一方也不发送。
两个方向需要分别关闭。
```

### TIME-WAIT

```text
主动关闭方发送最后 ACK 后进入 TIME-WAIT，等待 2MSL。
```

作用：

```text
保证最后 ACK 能到达。
让旧连接报文段从网络中消失，避免干扰新连接。
```

## 10. 简答题模板库

### TCP 如何保证可靠传输

```text
TCP 在不可靠的 IP 层之上，通过序号、确认、重传和滑动窗口实现可靠传输。
序号用于标识字节顺序，确认用于告知发送方哪些数据已正确收到。
当报文段丢失或确认超时时，TCP 进行超时重传；收到多个重复 ACK 时可进行快重传。
TCP 还使用校验和检测差错，并通过累计确认、SACK 等机制提高重传效率。
```

### 流量控制和拥塞控制区别

```text
流量控制用于防止发送方发送太快，使接收方来不及接收，主要依据接收方通告窗口 rwnd。
拥塞控制用于防止过多数据注入网络导致网络过载，主要依据拥塞窗口 cwnd、丢包和 RTT 等网络状态。
实际发送窗口通常取 min(rwnd, cwnd)。
```

### 为什么三次握手不是两次

```text
三次握手用于确认双方的发送和接收能力，并交换初始序号。
如果只有两次握手，已经失效的旧连接请求报文段可能延迟到达服务器，使服务器误以为客户端要建立新连接，造成资源浪费。
第三次握手可以让服务器确认客户端收到了自己的确认和初始序号，从而可靠建立连接。
```

### 为什么四次挥手通常是四次

```text
TCP 是全双工通信，两个方向的数据传输需要分别关闭。
一方发送 FIN 只表示自己没有数据要发送了，但对方可能还有数据要继续发送。
因此对方先返回 ACK，等自己的数据发送完后再发送 FIN，最后由主动关闭方返回 ACK。
```
