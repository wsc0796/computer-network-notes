## 2. 运输层高频简答模板

### 2.1 运输层的作用是什么

标准答法：

```text
运输层位于应用层和网络层之间，向应用进程提供端到端的逻辑通信服务。
网络层主要解决主机到主机的通信，运输层在此基础上通过端口号实现进程到进程的通信。
运输层还可以提供复用分用、差错检测，以及可靠传输、流量控制、拥塞控制和连接管理等功能。
```

速记：

```text
网络层找主机，运输层找进程。
```

### 2.2 为什么运输层需要端口

标准答法：

```text
一台主机上可能同时运行多个应用进程，仅靠 IP 地址只能定位到主机，不能定位到具体应用进程。
端口号用于标识主机中的应用进程，使运输层能够把收到的数据交付给正确的进程。
```

关键词：

```text
IP 地址定位主机，端口号定位进程。
```

### 2.3 UDP 的主要特点是什么

标准答法：

```text
UDP 是无连接的运输层协议，使用前不需要建立连接。
UDP 提供尽最大努力交付，不保证可靠、不保证按序、不进行流量控制和拥塞控制。
UDP 面向报文，保留应用层报文边界。
UDP 首部固定 8 字节，开销小、效率高，适合实时应用或简单请求响应场景。
```

### 2.4 TCP 的主要特点是什么

标准答法：

```text
TCP 是面向连接的可靠运输层协议。
TCP 提供全双工通信，面向字节流，通过序号、确认、重传、滑动窗口等机制保证可靠、有序交付。
TCP 还具有流量控制、拥塞控制和连接管理功能。
TCP 首部最小 20 字节。
```

### 2.5 UDP 和 TCP 的本质区别是什么

标准答法：

```text
UDP 无连接、不可靠、面向报文，首部小、开销低、效率高。
TCP 面向连接、可靠、面向字节流，具有确认、重传、流量控制、拥塞控制和连接管理机制。
因此 UDP 适合实时性强、可容忍丢包的场景，TCP 适合要求可靠交付的场景。
```

易错句：

```text
TCP 面向字节流，UDP 面向报文。
不要背反。
```

### 2.6 TCP 的序号和确认号怎么理解

标准答法：

```text
TCP 是面向字节流的协议，因此序号是对字节编号。
TCP 报文段中的序号 seq 表示本报文段所携带数据的第一个字节的编号。
确认号 ack 表示接收方期望收到的下一个字节编号。
若 ACK = x，表示 x 之前的所有字节都已经正确收到，下一步希望收到 x 号字节。
```

## 3. Karn 算法：重传后的 RTT 二义性

### 3.1 问题

TCP 需要根据 RTT 估计超时重传时间 RTO。

但如果一个报文段超时后被重传，后来收到了确认，发送方无法判断：

```text
这个 ACK 是确认原始报文段？
还是确认重传报文段？
```

这就是 RTT 样本的二义性。

### 3.2 Karn 算法

Karn 算法规定：

```text
凡是被重传过的报文段，其确认不能用于更新 RTT 样本。
```

原因：

```text
重传后的 ACK 无法确定对应哪一次发送，直接拿来算 RTT 会污染 RTO 估计。
```

### 3.3 修正 Karn 算法

如果完全不更新 RTO，网络状态变化时又可能不合适。

因此修正规则是：

```text
每发生一次超时重传，就把 RTO 加倍。
```

常见写法：

```text
新的 RTO = 2 * 旧的 RTO
```

等到之后收到没有被重传过的报文段的确认，再恢复正常 RTT 估计。

## 4. SACK 选择确认

### 4.1 为什么需要 SACK

普通累计确认只能告诉发送方：

```text
从哪个字节开始还没连续收到。
```

但它不能告诉发送方：

```text
后面哪些乱序到达的数据块其实已经收到了。
```

例如接收方已经收到：

```text
1-1000
1501-3000
3501-4500
```

缺失：

```text
1001-1500
3001-3500
```

普通累计确认可能只能反复返回：

```text
ACK = 1001
```

发送方可能误以为后面都没收到，造成不必要重传。

### 4.2 SACK 做了什么

SACK 允许接收方在 TCP 选项中告诉发送方：

```text
哪些不连续的数据块已经收到。
```

这样发送方只重传真正缺失的数据块。

考试答法：

```text
SACK 即选择确认，用于让接收方告知发送方已经收到的非连续数据块边界，
从而使发送方只重传缺失部分，减少不必要的重复传输，提高传输效率。
```

## 5. 零窗口问题与持续计时器

### 5.1 零窗口问题

TCP 流量控制中，接收方通过接收窗口 `rwnd` 告诉发送方自己还能接收多少数据。

如果接收方缓存满了，会通告：

```text
rwnd = 0
```

发送方收到后停止发送。

问题是：

```text
如果接收方后来缓存空出来，发送窗口更新报文却丢失，
发送方会一直以为 rwnd = 0，双方陷入等待。
```

### 5.2 持续计时器

解决方法：

```text
发送方收到零窗口通知后，启动持续计时器。
计时器到期时，发送方发送零窗口探测报文。
接收方收到探测后，回复当前窗口大小。
```

如果窗口仍为 0：

```text
继续等待并重启持续计时器。
```

如果窗口大于 0：

```text
恢复数据发送。
```

考试关键词：

```text
零窗口探测，防止窗口更新报文丢失造成死锁。
```

## 6. Nagle 算法与糊涂窗口综合征

### 6.1 糊涂窗口综合征

糊涂窗口综合征指：

```text
TCP 每次只发送很少几个字节的小报文段，导致首部开销很大，传输效率很低。
```

常见原因：

1. 发送方应用进程每次只交付少量数据。
2. 接收方应用进程每次只从缓存中取走少量数据，导致接收方频繁通告很小窗口。

### 6.2 Nagle 算法

Nagle 算法用于抑制发送方频繁发送小报文段。

核心思想：

```text
当连接中已经有未确认的小报文段时，发送方暂时不继续发送新的小报文段，
而是等待确认到达，或者等待积累到足够多的数据后再发送。
```

一句话记：

```text
有未确认小包时，先攒一攒，不要一个字节一个字节发。
```

### 6.3 接收方侧的改进

接收方不要刚释放一点点缓存就立刻通告很小窗口，而是等到：

```text
缓存空间足够大，或达到合适阈值后再通告。
```

## 7. AQM/RED 主动队列管理

### 7.1 尾部丢弃的问题

传统路由器常使用尾部丢弃：

```text
队列满了以后，后到达的分组直接丢弃。
```

问题：

```text
多个 TCP 连接可能同时发现丢包，同时降低发送速率；
随后又同时增加发送速率，导致网络流量同步振荡。
```

这种现象叫：

```text
全局同步。
```

### 7.2 AQM 的思想

AQM 是 Active Queue Management，主动队列管理。

思想：

```text
队列还没有满时，就提前随机丢弃或标记部分分组，
让发送方提前感知拥塞并降低发送速率。
```

### 7.3 RED

RED 是 Random Early Detection，随机早期检测。

基本作用：

```text
根据队列平均长度提前随机丢弃或标记分组，
尽量避免队列溢出和全局同步。
```

注意：

```text
RED 是典型 AQM 思想，但实际效果并非总是理想。
```

## 8. TIME-WAIT 与 2MSL

### 8.1 TIME-WAIT 在哪里出现

TCP 四次挥手中，主动关闭方发送最后一个 ACK 后，不会马上关闭，而是进入：

```text
TIME-WAIT
```

等待时间通常是：

```text
2MSL
```

MSL 是 Maximum Segment Lifetime，最长报文段寿命。

### 8.2 为什么要等 2MSL

原因一：保证最后一个 ACK 能到达对方。

```text
如果最后一个 ACK 丢失，被动关闭方会重传 FIN。
主动关闭方仍处于 TIME-WAIT，就可以再次发送 ACK。
```

原因二：让旧连接中的报文段从网络中消失。

```text
等待 2MSL 可以使本连接中滞留在网络里的旧报文段失效，
避免它们干扰之后使用相同四元组的新连接。
```

考试答法：

```text
TIME-WAIT 的作用是保证 TCP 全双工连接可靠释放，
并防止旧连接报文段影响后续新连接。
```

## 9. 公式速查卡

### 9.1 RTT 加权平均

```text
新的 RTTs = (1 - alpha) * 旧 RTTs + alpha * 新 RTT 样本
```

常见取值：

```text
alpha = 1/8
```

含义：

```text
旧估计值占主要权重，新样本用于逐步修正。
```

### 9.2 RTT 偏差

```text
新的 RTTd = (1 - beta) * 旧 RTTd + beta * |RTTs - 新 RTT 样本|
```

常见取值：

```text
beta = 1/4
```

### 9.3 RTO

```text
RTO = RTTs + 4 * RTTd
```

记忆：

```text
平均 RTT + 4 倍波动量。
网络波动越大，RTO 应适当变大。
```

### 9.4 发送窗口

```text
发送窗口上限 = min(rwnd, cwnd)
```

含义：

```text
rwnd 限制接收方能不能吃下。
cwnd 限制网络会不会堵。
```

### 9.5 发送速率近似

```text
rate = cwnd / RTT
```

### 9.6 拥塞时常见处理

超时重传：

```text
ssthresh = cwnd / 2
cwnd = 1 MSS
重新慢开始
```

3 个重复 ACK：

```text
快重传 + 快恢复
ssthresh = cwnd / 2
cwnd = ssthresh
进入拥塞避免
```

## 10. 高频易错点

1. TCP 是面向字节流，UDP 是面向报文。
2. UDP 不提供可靠服务，但可以由应用层自己实现可靠机制。
3. TCP 的序号按字节编号，不是按报文段编号。
4. `ACK = x` 表示 x 之前都收到了，下一步要 x。
5. 流量控制看接收方缓存，拥塞控制看网络负载。
6. 实际发送窗口是 `min(rwnd, cwnd)`。
7. 慢开始是指数增长，不是增长得慢。
8. 拥塞避免是线性增长。
9. 超时重传通常说明拥塞较严重，`cwnd` 降到 1。
10. 3 个重复 ACK 触发快重传，通常配合快恢复。
11. 三次握手用于确认双方收发能力、交换初始序号、防止旧连接请求。
12. 四次挥手是因为 TCP 全双工通信，两个方向要分别关闭。
13. TIME-WAIT 等待 2MSL，用于可靠释放和清除旧报文。
14. 零窗口不是连接断开，而是接收方暂时没有缓存空间。
15. SACK 不是新的确认号机制，而是在 TCP 选项中补充说明已收到的乱序块。

# 3. 需要补强的两个易错点

## 3.1 TCP/UDP面向对象不要背反

考试陷阱：

```text
TCP是面向报文的，UDP是面向字节流的。（错）
```

正确说法：

```text
TCP是面向字节流的。
UDP是面向报文的。
```

为什么：

```text
TCP把应用数据看成连续字节流，不保留应用层消息边界。
UDP一次发送一个报文，接收方一次收一个完整UDP报文，保留消息边界。
```

一句话记：

```text
TCP像水管，按字节流动；UDP像快递，一件一件报文投递。
```

## 3.2 网络层服务类型 vs 运输层运行机制

作业判断题：

```text
网络层提供数据报或虚电路服务不影响运输层的运行机制，但提供不同的服务质量。
```

答案：

```text
对。
```

考试答法：

```text
运输层协议运行在端系统中，依靠端系统的TCP/UDP机制完成进程到进程通信；
网络层在下面提供主机到主机的通信服务，可能是数据报服务，也可能是虚电路服务。
网络层服务类型不会改变运输层协议本身的基本运行机制，
但会影响运输层感受到的丢包、时延、乱序、吞吐率等服务质量。
```

记忆：

```text
机制看运输层自己；质量受网络层影响。
```
