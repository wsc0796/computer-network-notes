## 0. 本章主线

运输层位于应用层和网络层之间。网络层提供的是主机到主机的尽力而为交付，运输层在此基础上向应用进程提供端到端通信服务。

本章可以按这条线背：

1. 网络层服务不可靠，所以运输层要解决进程到进程通信、复用/分用、差错检测，以及在需要时提供可靠传输、流量控制、拥塞控制和连接管理。
2. 端口号用于在一台主机内定位具体应用进程。
3. UDP 是无连接、尽最大努力、面向报文、首部小的运输层协议。
4. TCP 是面向连接、可靠、全双工、面向字节流的运输层协议。
5. UDP 适合快、简单、可容忍丢包或由应用层自己保证可靠性的场景；TCP 适合要求可靠、有序、无差错交付的场景。

## 1. 运输层要解决的问题

### 1.1 进程之间的通信

网络层解决的是主机到主机通信，运输层解决的是应用进程到应用进程通信。

常见表述：

- IP 地址用于找到目的主机。
- 端口号用于在目的主机中找到具体应用进程。
- 运输层在两个运输层实体之间建立端到端的逻辑通信信道。

### 1.2 最低限度服务

运输层至少提供两类基本服务：

1. 进程到进程的数据交付。
2. 差错检测。

注意区分：

- 网络层 IP 首部校验和只检查 IP 首部，不检查数据部分。
- 运输层校验和检查运输层报文及相关信息，能覆盖首部和数据。
- 数据链路层校验主要检测链路传输中的比特错误。

### 1.3 增强服务

运输层可以在网络层尽力而为服务之上提供增强功能：

- 可靠传输：确认、序号、重传、检错等。
- 流量控制：避免发送方发送太快，把接收方缓存撑爆。
- 拥塞控制：避免网络内部过载。
- 连接管理：建立连接、释放连接、维护状态。

### 1.4 TPDU

两个对等运输实体通信时传送的数据单位叫运输协议数据单元，简称 `TPDU`。

- TCP 传送的数据单位：`TCP 报文段 segment`
- UDP 传送的数据单位：`UDP 报文` 或 `用户数据报`

## 2. 端口与套接字

### 2.1 端口的作用

端口是通信的抽象终点，用来标识本机应用层中的进程。运输层协议把数据交给指定端口，操作系统再把数据交给对应进程。

端口的特点：

- 端口号是 16 位。
- 理论范围：`0-65535`。
- 端口号只具有本地意义。
- 不同主机上的相同端口号之间没有必然联系。

### 2.2 端口分类

| 类型 | 范围 | 用途 |
|---|---:|---|
| 熟知端口/系统端口 | `0-1023` | 公共域协议使用，如 HTTP、DNS 等 |
| 登记端口 | `1024-49151` | 没有熟知端口号的应用可向 IANA 登记 |
| 短暂端口/动态端口 | `49152-65535` | 客户进程临时选择使用，通信结束后释放 |

### 2.3 常见协议与传输层

| 应用协议 | 常见运输层协议 | 记忆点 |
|---|---|---|
| DNS | UDP 为主 | 单次请求/响应，快 |
| DHCP | UDP | 主机配置 |
| RIP | UDP | 路由选择协议 |
| TFTP | UDP | 简单文件传输 |
| HTTP/HTTPS | TCP | 可靠传输 |
| SMTP | TCP | 电子邮件发送 |
| FTP | TCP | 文件传输 |
| TELNET | TCP | 远程登录 |

### 2.4 Socket 的标识

UDP 套接字用二元组标识：

```text
<IP 地址, 端口号>
```

因此，目的 IP 地址和目的端口号相同的 UDP 报文会交付给同一个套接字，和源 IP、源端口无关。

TCP 连接套接字用四元组标识：

```text
<源 IP 地址, 源端口号, 目的 IP 地址, 目的端口号>
```

一个 TCP 服务器为了同时服务多个客户，通常使用监听套接字和多个连接套接字。每个连接套接字只和一个客户通信。

## 3. UDP

### 3.1 UDP 概述

UDP 建立在 IP 的尽力而为服务之上，只增加了少量功能：

1. 复用和分用。
2. 差错检测。

网络层尽力而为服务的含义：

- 尽最大努力把数据包交付到目的主机。
- 不保证可靠性和顺序。
- 不保证带宽和时延。

### 3.2 UDP 的主要特点

| 特点 | 说明 |
|---|---|
| 无连接 | 发送数据前不需要建立连接 |
| 尽最大努力交付 | 不保证可靠交付 |
| 面向报文 | 应用层交下来的报文不合并、不拆分，添加 UDP 首部后直接交给 IP |
| 无拥塞控制 | 不会因为网络拥塞主动降低源主机发送速率 |
| 支持多种通信方式 | 一对一、一对多、多对一、多对多 |
| 首部开销小 | UDP 首部只有 8 字节 |

### 3.3 为什么需要 UDP

UDP 的价值在于简单和快：

- 没有建立连接的时延。
- 不进行拥塞控制和流量控制，不限制发送速率。
- 报头开销小。
- 协议处理简单。

适合 UDP 的应用：

- 容忍丢包但对延迟敏感：IP 电话、视频会议、多媒体应用。
- 以单次请求/响应为主：DNS、RIP、DHCP。
- 应用层自己实现可靠性：可以基于 UDP 自行确认、重传、排序。

### 3.4 UDP 面向报文

发送方：

- UDP 对应用层交下来的报文既不合并，也不拆分。
- 添加 UDP 首部后，按原样交给 IP 层。

接收方：

- UDP 去掉首部后，把完整报文一次性交给应用层。

考试常问：

- UDP 保留应用层报文边界。
- 应用程序必须选择合适大小的报文。
- 报文太长：IP 层可能分片，降低效率，丢一个分片就可能导致整个报文失败。
- 报文太短：首部相对占比过大，降低效率。

### 3.5 UDP 首部

UDP 首部固定 8 字节，4 个字段，每个字段 2 字节。

| 字段 | 长度 | 作用 |
|---|---:|---|
| 源端口 | 2 字节 | 需要对方回信时使用；不需要时可全 0 |
| 目的端口 | 2 字节 | 终点交付报文时必须使用 |
| 长度 | 2 字节 | UDP 用户数据报长度，最小值为 8 |
| 校验和 | 2 字节 | 检测 UDP 用户数据报传输中是否有错；首部和数据一起参与计算 |

目的端口错误时：

- 接收方 UDP 发现目的端口号没有对应应用进程，会丢弃该报文。
- 通常由 ICMP 向发送方返回“端口不可达”差错报文。

## 4. TCP

### 4.1 TCP 概述

TCP 是面向连接的运输层协议。它在无连接、不可靠的 IP 网络服务基础上，通过确认、序号、重传、检错等机制，向应用层提供可靠交付服务。

### 4.2 TCP 的主要特点

| 特点 | 说明 |
|---|---|
| 面向连接 | 通信前建立连接，通信后释放连接 |
| 点对点 | 每条 TCP 连接只能有两个端点 |
| 可靠交付 | 通过序号、确认、重传、检错等保证可靠性 |
| 全双工通信 | 双方可以同时发送和接收数据 |
| 面向字节流 | 不保留应用层报文边界 |

TCP 的“连接”不是物理线路独占，而是逻辑连接。IP 数据报仍然可能乱序、丢失或重复，但 TCP 在端系统中通过协议机制把它组织成可靠的字节流。

### 4.3 面向连接的三种理解

| 类型 | 可靠性来源 | 特点 |
|---|---|---|
| 面向连接的电路交换 | 物理层保证可靠 | 双方独占物理连接，数据按序发送和接收 |
| 面向连接的虚电路 | 网络层保证可靠 | 采用复用技术，分组按序发送和接收 |
| 面向连接的 TCP | 运输层协议保证可靠 | 采用确认、序号、重传等，构造全双工可靠逻辑信道 |

### 4.4 TCP 面向字节流

TCP 把应用层交下来的数据看作连续的字节流。

重点：

- 发送的字节流和接收的字节流完全一致。
- TCP 不保留应用层报文边界。
- 应用层如果需要区分消息边界，要自己设计格式，例如长度字段、分隔符等。

### 4.5 TCP 报文段与首部

TCP 传送的数据单位叫报文段。一个 TCP 报文段分为首部和数据两部分。

TCP 首部：

- 前 20 字节是固定首部。
- 后面可以有选项，长度为 `4n` 字节。
- TCP 首部最小长度是 20 字节。
- 数据偏移字段表示 TCP 报文段的数据起始处距离报文段起始处有多远，也就是 TCP 首部长度，以 4 字节为单位。

### 4.6 TCP 首部字段

| 字段 | 作用 |
|---|---|
| 源端口、目的端口 | 各 2 字节，是运输层与应用层的服务接口 |
| 序号 seq | 本报文段数据载荷中第一个字节在字节流中的序号 |
| 确认号 ack | 期望收到对方下一个报文段数据第一个字节的序号 |
| 数据偏移 | TCP 首部长度，以 4 字节为单位 |
| 窗口 | 接收方告诉发送方自己还能接收多少数据，用于流量控制 |
| 校验和 | 检测 TCP 报文段传输中是否出错 |
| 紧急指针 | 与 URG 配合，指出紧急数据的末尾位置 |
| 选项 | 长度可变，最长可达 40 字节 |
| 填充 | 使 TCP 首部长度成为 4 字节的整数倍 |

### 4.7 TCP 标志位

| 标志位 | 含义 | 典型用途 |
|---|---|---|
| URG | 紧急 | `URG=1` 时紧急指针有效，提示尽快传送紧急数据 |
| ACK | 确认 | `ACK=1` 时确认号字段有效 |
| PSH | 推送 | 接收方应尽快交付应用进程，不必等缓存填满 |
| RST | 复位 | TCP 连接出现严重错误，释放连接后重新建立 |
| SYN | 同步 | 建立连接时同步序号 |
| FIN | 终止 | 发送方数据发送完毕，请求释放连接 |

SYN 的两个常考组合：

- `SYN=1, ACK=0`：连接请求报文段。
- `SYN=1, ACK=1`：连接接受报文段。

FIN：

- `FIN=1` 表示此报文段发送端的数据已发送完毕，并要求释放运输连接。

### 4.8 TCP 选项

| 选项 | 作用 |
|---|---|
| MSS | 最大报文段长度，指 TCP 报文段中数据字段的最大长度，与接收窗口值无关 |
| Window Scale | 窗口扩大选项，建立连接时协商窗口比例因子 |
| Timestamp | 时间戳选项，占 10 字节，用于计算 RTT 和防止序号绕回 PAWS |

MSS 重点：

- MSS 是 TCP 数据部分的最大长度，不包括 TCP 首部、IP 首部和链路层首尾部。
- 建立连接时，每个主机可以声明自己能够接收的 MSS。
- 默认值常见为 536 字节。
- MSS 不宜太小：头部占比大，网络利用率低。
- MSS 不宜太大：IP 层可能分片，开销增大，分片出错会导致整个分组失败。
- 应尽量让 IP 层传输时不再分片。

窗口扩大：

```text
实际接收窗口大小 = window size * 2^window scale
```

时间戳主要功能：

- 计算往返时间 RTT。
- 防止序号绕回 PAWS：在高速网络中，32 位序号空间可能很快用完，时间戳可帮助区分新报文段和迟到很久的旧报文段。

## 5. 可靠传输复习补充

课程截图目录出现了可靠传输相关内容，以下是复习时必须掌握的标准知识。

### 5.1 可靠传输要解决什么

不可靠信道可能导致：

- 比特差错。
- 分组丢失。
- 分组重复。
- 分组失序。

可靠传输常用机制：

- 差错检测：发现出错。
- 确认 ACK：接收方告诉发送方已经正确收到。
- 超时重传：发送方等待太久没有确认就重传。
- 序号：区分新数据和重复数据，按序交付。
- 滑动窗口：允许连续发送多个报文段，提高效率。

### 5.2 累计确认

TCP 的确认号表示“期望收到的下一个字节序号”。

例如：

```text
确认号 = 1001
```

含义：序号 1000 之前的字节都已经正确收到，下一步希望收到 1001 开始的数据。

## 6. TCP 流量控制复习补充

流量控制解决的是发送方和接收方之间的速度匹配问题。

核心：

- 接收方通过 TCP 首部中的窗口字段告诉发送方自己的接收能力。
- 发送方根据接收窗口限制已发送但未确认的数据量。
- 防止发送方发送过快，导致接收方缓存溢出。

容易和拥塞控制混淆：

- 流量控制看接收方是否来得及接收。
- 拥塞控制看网络内部是否拥塞。

## 7. TCP 拥塞控制复习补充

拥塞控制解决的是网络内部过载问题。

常见变量：

- `cwnd`：拥塞窗口，由发送方根据网络拥塞情况维护。
- `ssthresh`：慢开始门限。

常见算法：

| 阶段 | 特点 |
|---|---|
| 慢开始 | `cwnd` 从小开始，按指数增长 |
| 拥塞避免 | 到达门限后，`cwnd` 线性增长 |
| 快重传 | 收到多个重复 ACK 时，不等超时就重传 |
| 快恢复 | 快重传后不直接回到 1，而是调整窗口继续传输 |

发送窗口通常受两方面限制：

```text
发送窗口 = min(接收窗口 rwnd, 拥塞窗口 cwnd)
```

## 8. TCP 连接管理复习补充

### 8.1 三次握手

建立 TCP 连接通常使用三次握手。

```text
客户端 -> 服务器：SYN=1, seq=x
服务器 -> 客户端：SYN=1, ACK=1, seq=y, ack=x+1
客户端 -> 服务器：ACK=1, seq=x+1, ack=y+1
```

为什么需要三次握手：

- 确认双方发送能力和接收能力都正常。
- 同步双方初始序号。
- 防止历史连接请求报文导致错误建立连接。

### 8.2 四次挥手

释放 TCP 连接通常使用四次挥手。

```text
主动关闭方 -> 被动关闭方：FIN=1
被动关闭方 -> 主动关闭方：ACK=1
被动关闭方 -> 主动关闭方：FIN=1
主动关闭方 -> 被动关闭方：ACK=1
```

为什么通常是四次：

- TCP 是全双工通信。
- 一方没有数据要发了，不代表另一方也没有数据要发。
- 因此关闭两个方向的数据流通常要分别确认。

TIME_WAIT 的作用：

- 保证最后一个 ACK 能到达对方。
- 让旧连接中的迟到报文在网络中自然消失，避免影响新连接。

## 9. UDP 与 TCP 对比

| 对比项 | UDP | TCP |
|---|---|---|
| 是否连接 | 无连接 | 面向连接 |
| 可靠性 | 尽最大努力交付，不保证可靠 | 可靠交付 |
| 数据形式 | 面向报文 | 面向字节流 |
| 报文边界 | 保留应用层报文边界 | 不保留报文边界 |
| 首部长度 | 固定 8 字节 | 最小 20 字节，可变 |
| 通信方式 | 一对一、一对多、多对一、多对多 | 点对点 |
| 流量控制 | 无 | 有 |
| 拥塞控制 | 无 | 有 |
| 典型应用 | DNS、DHCP、RIP、视频会议 | HTTP、HTTPS、FTP、SMTP、TELNET |

## 10. 高频简答模板

### 10.1 运输层的作用是什么？

运输层在网络层提供的主机到主机通信基础上，为应用进程提供进程到进程的端到端通信服务。它通过端口实现复用和分用，并提供差错检测；在 TCP 中还提供可靠传输、流量控制、拥塞控制和连接管理等增强服务。

### 10.2 为什么运输层需要端口？

IP 地址只能定位到主机，不能定位主机中的具体应用进程。端口号用于标识本机应用层中的进程，运输层通过端口把收到的数据交付给正确的应用进程。

### 10.3 UDP 的主要特点是什么？

UDP 是无连接、尽最大努力交付、面向报文的运输层协议。它不提供可靠传输、流量控制和拥塞控制，支持一对一、一对多、多对一和多对多通信，首部只有 8 字节，协议处理简单，适合对实时性要求高或以单次请求/响应为主的应用。

### 10.4 TCP 的主要特点是什么？

TCP 是面向连接、点对点、可靠、全双工、面向字节流的运输层协议。它在不可靠的 IP 服务之上，通过序号、确认、重传、校验、流量控制和拥塞控制等机制，为应用层提供可靠交付服务。

### 10.5 UDP 和 TCP 的本质区别是什么？

UDP 追求简单和低开销，不保证可靠交付，保留报文边界；TCP 追求可靠、有序、无差错交付，建立连接并以字节流方式传输，不保留应用层报文边界。

### 10.6 TCP 的序号和确认号怎么理解？

TCP 对字节流中的每个字节编号。序号表示本报文段数据部分第一个字节的编号；确认号表示接收方期望收到的下一个字节编号，因此确认号具有累计确认含义。

## 11. 易错点速记

- 运输层通信对象是进程，不是主机。
- IP 地址找主机，端口号找进程。
- UDP 首部固定 8 字节，TCP 首部最小 20 字节。
- UDP 面向报文，TCP 面向字节流。
- UDP 不合并、不拆分应用层报文；TCP 不保留应用层报文边界。
- UDP 套接字用二元组 `<IP, port>` 标识。
- TCP 连接用四元组 `<源IP, 源端口, 目的IP, 目的端口>` 标识。
- `ACK=1` 时确认号才有效。
- `SYN=1, ACK=0` 是连接请求；`SYN=1, ACK=1` 是连接接受。
- `FIN=1` 表示发送端数据发送完毕，请求释放连接。
- MSS 是 TCP 数据字段最大长度，不是整个帧长度，也和接收窗口值无关。
- 流量控制管接收方，拥塞控制管网络。

# 计算机网络 - 运输层笔记（3-4）

## 5.5 TCP 可靠传输的实现

TCP 要在不可靠的 IP 层之上提供可靠传输，核心手段是：

1. 以字节为单位编号。
2. 使用滑动窗口进行连续发送。
3. 使用确认机制判断数据是否到达。
4. 使用超时重传处理丢失。
5. 必要时使用选择确认 SACK 提高重传效率。

### 5.5.1 以字节为单位的滑动窗口

TCP 的滑动窗口不是按“报文段”编号，而是按“字节”编号。TCP 把应用层交下来的字节流看成连续的数据序列，每个字节都有序号。

发送方 A 和接收方 B 分别维护：

| 窗口 | 含义 |
|---|---|
| 发送窗口 | 发送方当前允许发送的一段字节范围 |
| 接收窗口 | 接收方当前还能接收的一段字节范围 |

发送窗口中的数据可以分为几类：

| 区域 | 含义 |
|---|---|
| 已发送并已确认 | 可以从发送缓存中删除 |
| 已发送但未确认 | 不能删除，可能需要重传 |
| 允许发送但尚未发送 | 可以继续发送 |
| 不允许发送 | 超出当前窗口范围，暂时不能发 |

接收窗口中的数据也有类似状态：

| 区域 | 含义 |
|---|---|
| 已按序接收并交付应用层 | 可以释放缓存 |
| 已接收但还不能交付 | 前面有缺失字节，只能暂存 |
| 允许接收 | 落在接收窗口内 |
| 不允许接收 | 不在接收窗口范围内，通常丢弃 |

#### 累计确认

TCP 通常使用累计确认。确认号 `ACK = x` 表示：

> 到序号 `x - 1` 为止的所有字节都已经正确收到，下一个期望收到的字节序号是 `x`。

例如确认号为 `31`，表示 30 号及之前的字节已经全部收到，接收方下一步希望收到 31 号字节。

#### 窗口滑动

发送方收到新的确认后，发送窗口左边界右移；接收方成功按序接收数据后，接收窗口也会随之移动。

窗口是否能继续推进，关键看：

1. 是否收到新的确认。
2. 接收方是否还有缓存空间。
3. 网络是否允许继续发送，后面由拥塞控制决定。

### 5.5.2 超时重传时间的选择

TCP 发送报文段后，如果在规定时间内没有收到确认，就会认为该报文段可能丢失，需要重传。

超时重传时间不能设置得太短，也不能太长：

| RTO 设置 | 后果 |
|---|---|
| 太短 | 报文可能只是延迟，发送方却误以为丢失，造成不必要重传 |
| 太长 | 真正丢失后迟迟不重传，降低传输效率 |

因此 TCP 需要根据网络实际情况动态估计 RTT。

#### 加权平均往返时间 RTTs

RTT 表示从发送一个报文段到收到对应确认的时间。

TCP 使用加权平均 RTT：

```text
新的 RTTs = (1 - alpha) * 旧的 RTTs + alpha * 新的 RTT 样本
```

常见取值：

```text
alpha = 1/8 = 0.125
```

含义：旧 RTTs 占主要权重，新样本用于逐步修正估计值。

#### 超时重传时间 RTO

RTO 表示 Retransmission Time-Out，超时重传时间。

RFC 6298 的常见公式：

```text
RTO = RTTs + 4 * RTTd
```

其中 RTTd 表示 RTT 偏差的加权平均：

```text
新的 RTTd = (1 - beta) * 旧的 RTTd + beta * |RTTs - 新 RTT 样本|
```

常见取值：

```text
beta = 1/4 = 0.25
```

考试要点：

1. `RTTs` 反映平均 RTT。
2. `RTTd` 反映 RTT 波动。
3. 网络波动越大，`RTTd` 越大，RTO 也应适当变大。

#### Karn 算法

TCP 中有一个二义性问题：如果某个报文段超时后被重传，后来收到了确认，发送方无法判断这个确认对应的是原始报文段，还是重传报文段。

因此 Karn 算法规定：

> 计算 RTT 时，只要报文段被重传过，就不要用它的确认来更新 RTT 样本。

但如果完全不更新，RTO 可能长期不合适。因此修正的 Karn 算法还规定：

> 报文段每超时重传一次，就把 RTO 加倍。

常见写法：

```text
新的 RTO = gamma * 旧的 RTO
```

其中通常：

```text
gamma = 2
```

等网络恢复、再次收到未重传报文段的确认后，再继续正常更新 RTT。

### 5.5.3 选择确认 SACK

普通累计确认只能告诉发送方“从哪里开始还缺数据”，但不能精确告诉中间哪些数据已经收到。

例如：

```text
已收到：1-1000，1501-3000，3501-4500
缺失：1001-1500，3001-3500
```

如果只使用累计确认，接收方可能只能反复确认 `ACK = 1001`，发送方不清楚后面哪些字节已经收到，可能重复发送已经到达的数据。

SACK 的作用：

> 让接收方告诉发送方“哪些不连续的数据块已经收到了”，发送方只重传缺失的数据块。

SACK 可以减少不必要重传，提高网络利用率。

## 5.6 TCP 的流量控制

流量控制解决的问题是：

> 发送方不要发得太快，避免接收方来不及接收。

注意：流量控制关注的是接收方能力，不是网络拥塞情况。

### 5.6.1 利用滑动窗口实现流量控制

接收方通过 TCP 报文段首部中的窗口字段，把自己的可用接收缓存大小告诉发送方。

这个窗口叫：

```text
rwnd = receiver window，接收窗口
```

发送方必须让发送窗口不超过接收方通告的 `rwnd`。

#### 零窗口问题

如果接收方缓存满了，会通告：

```text
rwnd = 0
```

发送方此时不能继续发送普通数据。

问题在于：如果接收方后来缓存空出来，发送了“窗口非零”的通知，但该通知丢失，发送方就会一直以为窗口为 0，双方陷入僵局。

解决方法：持续计时器。

#### 持续计时器

当发送方收到零窗口通知后，启动持续计时器。

计时器到期后，发送方发送一个零窗口探测报文段，询问接收方当前窗口大小。

接收方返回窗口通告：

1. 如果窗口仍为 0，发送方继续等待并重新启动计时器。
2. 如果窗口大于 0，发送方恢复发送。

### 5.6.2 TCP 的传输效率

TCP 既要可靠，也要高效。如果每次只发送很小的数据，首部开销会很大。

#### 糊涂窗口综合征

糊涂窗口综合征指：

> TCP 每次只发送一个或几个字节的小报文段，导致大量首部开销，传输效率极低。

可能来源有两个：

| 来源 | 表现 | 解决思路 |
|---|---|---|
| 发送方应用进程每次只交付很少数据 | 发送方频繁发送小报文段 | Nagle 算法 |
| 接收方应用进程每次只取走很少数据 | 接收方反复通告很小窗口 | 等窗口足够大再通告 |

#### Nagle 算法

Nagle 算法用于减少发送方产生的小报文段。

基本思想：

1. 若发送方已有未确认的小报文段，则先不继续发送新的小报文段。
2. 等收到前一个小报文段的确认，或者积累到足够多的数据后，再发送。

一句话记忆：

> 有未确认小包时，先攒一攒，不要一个字节一个字节发。

#### 接收方解决方法

接收方不要刚释放一点点缓存就马上通告很小的窗口，而是等待：

1. 接收缓存有足够空间。
2. 或者达到一个合适阈值。

这样可以避免发送方被诱导着发送大量小报文段。

## 5.7 / 5.8 TCP 拥塞控制

拥塞控制解决的问题是：

> 防止过多数据注入网络，使路由器或链路过载。

网络拥塞的表现：

1. 分组在网络中排队等待，时延增大。
2. 路由器缓存不足，出现丢包。
3. 发送方不断重传，进一步加重网络负担。
4. 吞吐量可能下降，严重时接近 0。

专业表述：

```text
拥塞产生原因：对网络资源的需求 > 可用资源
```

### 流量控制和拥塞控制的区别

| 对比项 | 流量控制 | 拥塞控制 |
|---|---|---|
| 作用范围 | 点对点、端到端的接收能力控制 | 全局网络负载控制 |
| 关注对象 | 接收方是否来得及接收 | 网络是否承受得住 |
| 主要依据 | 接收方通告窗口 `rwnd` | 拥塞窗口 `cwnd`、丢包、RTT 等 |
| 主动方 | 接收方告诉发送方窗口大小 | 发送方根据网络状态调整发送速率 |

### 开环控制和闭环控制

| 类型 | 含义 |
|---|---|
| 开环控制 | 事先设计好规则，运行中不根据反馈动态调整 |
| 闭环控制 | 检测网络状态，根据反馈调整发送策略 |

TCP 拥塞控制属于闭环控制：发现拥塞迹象后降低发送速率，网络恢复后再逐步提高。

### 拥塞窗口 cwnd

TCP 发送方维护一个拥塞窗口：

```text
cwnd = congestion window
```

发送速率近似为：

```text
rate = cwnd / RTT
```

发送方实际能发送多少数据，要同时受接收方窗口和拥塞窗口限制：

```text
发送窗口上限 = Min(rwnd, cwnd)
```

含义：

1. `rwnd < cwnd`：接收方能力限制发送速度。
2. `cwnd < rwnd`：网络拥塞控制限制发送速度。

### 拥塞控制的四个算法

RFC 5681 中常见四个算法：

1. 慢开始 slow start。
2. 拥塞避免 congestion avoidance。
3. 快重传 fast retransmit。
4. 快恢复 fast recovery。

### 慢开始

慢开始不是慢速增长，而是从小窗口开始指数增长。

初始时：

```text
cwnd = 1 MSS
```

每收到一个新的确认，`cwnd` 增加 1 个 MSS。

因此每经过一个 RTT，`cwnd` 大致翻倍：

```text
1, 2, 4, 8, 16, ...
```

慢开始需要配合慢开始门限：

```text
ssthresh = slow start threshold
```

规则：

| 条件 | 使用算法 |
|---|---|
| `cwnd < ssthresh` | 慢开始 |
| `cwnd > ssthresh` | 拥塞避免 |
| `cwnd = ssthresh` | 两者都可以 |

### 拥塞避免

拥塞避免采用线性增长。

常见表述：

> 每经过一个 RTT，cwnd 增加约 1 个 MSS。

所以增长趋势是：

```text
慢开始：指数增长
拥塞避免：线性增长
```

### 出现拥塞时的调整

如果发生超时重传，通常认为网络已经拥塞。

传统处理方式：

```text
ssthresh = cwnd / 2
cwnd = 1 MSS
重新进入慢开始
```

这体现了 TCP 的拥塞控制思想：

> 网络正常时逐步试探增加，发现拥塞时迅速降低发送速率。

### 快重传

快重传用于更快发现丢包，不必等到超时。

触发条件：

> 发送方连续收到 3 个重复确认。

含义：接收方一直在确认同一个序号，说明这个序号开始的数据很可能丢失，但后面的报文段已经到达。

发送方收到 3 个重复确认后，立即重传丢失的报文段。

### 快恢复

快恢复通常和快重传配合使用。

收到 3 个重复确认时，说明网络中仍有后续报文段到达，网络并非完全瘫痪，因此不必像超时那样把 `cwnd` 降到 1。

常见处理：

```text
ssthresh = cwnd / 2
cwnd = ssthresh
进入拥塞避免
```

考试记忆：

| 情况 | 处理 |
|---|---|
| 超时重传 | 拥塞更严重，`cwnd` 降为 1，重新慢开始 |
| 3 个重复 ACK | 认为只是个别丢失，快重传 + 快恢复 |

### 主动队列管理 AQM

传统路由器常采用尾部丢弃：

> 队列满后，后到达的分组直接丢弃。

尾部丢弃的问题：

1. 多个 TCP 连接可能同时发现丢包。
2. 它们同时降低发送速率，随后又同时增加。
3. 网络流量出现同步振荡，这叫全局同步。

主动队列管理 AQM 的思想：

> 队列还没有满时，就主动、随机地丢弃或标记部分分组，让发送方提前降低速率。

典型算法：RED，Random Early Detection，随机早期检测。

注意：RED 效果不总是理想，课程中提到多年实践后并未成为特别成功的方案。

## 5.8 TCP 的运输连接管理

TCP 是面向连接的协议，传输过程分为三个阶段：

1. 连接建立。
2. 数据传送。
3. 连接释放。

TCP 连接管理要解决：

1. 确认对方存在。
2. 协商初始序号等参数。
3. 分配缓存、连接表项等资源。
4. 在释放连接时保证数据可靠交付并回收资源。

### 5.8.1 TCP 连接建立：三次握手

TCP 连接采用客户-服务器方式。

| 角色 | 含义 |
|---|---|
| 主动发起连接 | 客户端 client |
| 被动等待连接 | 服务器 server |

三次握手过程：

1. 客户端发送 `SYN = 1`，选择初始序号 `seq = x`，进入 `SYN-SENT`。
2. 服务器收到后，回复 `SYN = 1, ACK = 1`，确认号 `ack = x + 1`，同时选择自己的初始序号 `seq = y`，进入 `SYN-RCVD`。
3. 客户端收到后，发送 `ACK = 1`，确认号 `ack = y + 1`，序号 `seq = x + 1`，进入 `ESTABLISHED`。服务器收到后也进入 `ESTABLISHED`。

记忆版：

```text
第一次：客户端请求连接
第二次：服务器同意并请求确认
第三次：客户端确认服务器的同意
```

#### 为什么需要三次握手

三次握手的关键目的：

1. 双方都确认对方的发送能力和接收能力正常。
2. 双方交换并确认初始序号。
3. 防止已经失效的连接请求报文段突然到达服务器，造成错误连接。

如果只有两次握手，旧的连接请求报文可能在网络中延迟很久后到达服务器，服务器误以为客户端要建立新连接，从而分配资源，造成浪费。

### 5.8.2 TCP 连接释放：四次挥手

TCP 连接释放比较复杂，因为 TCP 是全双工通信，两个方向的数据传输需要分别关闭。

四次挥手过程：

1. 主动关闭方发送 `FIN = 1, seq = u`，进入 `FIN-WAIT-1`。
2. 被动关闭方回复 `ACK = 1, ack = u + 1`，进入 `CLOSE-WAIT`；主动关闭方收到后进入 `FIN-WAIT-2`。
3. 被动关闭方数据发送完毕后，发送 `FIN = 1, ACK = 1, seq = v, ack = u + 1`，进入 `LAST-ACK`。
4. 主动关闭方回复 `ACK = 1, ack = v + 1`，进入 `TIME-WAIT`；被动关闭方收到后关闭。主动关闭方等待一段时间后关闭。

记忆版：

```text
第一次：我这边没数据要发了
第二次：知道了，但我可能还有数据要发
第三次：我也发完了，可以关闭
第四次：知道了，最终关闭
```

#### 为什么连接释放通常需要四次

TCP 是全双工通信。

一方发送 FIN 只表示：

> 我这个方向没有数据要发送了。

但另一方可能还有数据要继续发送，所以不能立即关闭两个方向。对方需要先 ACK，再等自己数据发送完后单独发送 FIN。

### TIME-WAIT 和 2MSL 等待

主动关闭方发送最后一个 ACK 后，不会立刻关闭，而是进入 `TIME-WAIT` 状态，等待 `2MSL`。

MSL 表示 Maximum Segment Lifetime，最长报文段寿命。

等待 `2MSL` 的原因：

1. 保证最后一个 ACK 能到达对方。如果 ACK 丢失，对方会重传 FIN，主动关闭方还能再次回复 ACK。
2. 让本连接中的旧报文段在网络中全部消失，避免影响后续使用相同四元组的新连接。

考试高频句：

> TIME-WAIT 的作用是保证 TCP 全双工连接可靠释放，并防止旧连接报文段干扰新连接。

## 高频易错点总结

1. TCP 滑动窗口以字节为单位，不是以报文段为单位。
2. `ACK = x` 表示期望收到序号为 `x` 的字节。
3. 流量控制看接收方缓存，拥塞控制看网络承载能力。
4. 实际发送窗口上限是 `Min(rwnd, cwnd)`。
5. 慢开始是指数增长，不是增长得慢。
6. 拥塞避免是线性增长。
7. 超时重传通常把 `cwnd` 降到 1，重新慢开始。
8. 3 个重复 ACK 触发快重传，通常进入快恢复。
9. 三次握手用于建立连接并防止旧连接请求造成错误连接。
10. 四次挥手是因为 TCP 是全双工通信，两个方向要分别关闭。
11. TIME-WAIT 等待 `2MSL`，不是随便等待，而是为了可靠释放和清除旧报文。
