## 5. DNS 题型

### 5.1 DNS 的作用

```text
DNS 用于将人容易记忆的域名解析为网络通信所需的 IP 地址。
```

注意区分：

```text
DNS：域名 -> IP
ARP：IP -> MAC
```

### 5.2 递归查询和迭代查询

递归查询：

```text
客户端把查询任务交给 DNS 服务器，由 DNS 服务器继续查询并最终返回完整结果。
```

迭代查询：

```text
DNS 服务器不一定直接返回最终答案，而是告诉查询者下一步应该问哪个 DNS 服务器。
```

### 5.3 DNS 简答模板

```text
DNS 是域名系统，用于将域名解析为 IP 地址。
主机通常先查询本地域名服务器；如果本地域名服务器没有缓存结果，
则继续向根域名服务器、顶级域名服务器、权威域名服务器查询，
最终获得目标主机 IP 地址并返回给客户端。
```

### 5.4 DNS 常见考法

| 考法 | 答题关键词 |
|---|---|
| DNS 作用 | 域名解析为 IP |
| DNS 用 TCP 还是 UDP | UDP 为主，TCP 可用 |
| 递归 vs 迭代 | 是否由被查询服务器负责查到底 |
| DNS 与 ARP 区别 | DNS 域名到 IP，ARP IP 到 MAC |

## 6. HTTP/HTTPS 题型

### 6.1 HTTP 基础

HTTP 是超文本传输协议，采用请求-响应模式。

常见特点：

```text
基于 TCP
默认端口 80
明文传输
无状态
```

### 6.2 为什么 HTTP 无状态

```text
HTTP 协议本身不会自动记住前后请求之间的用户状态。
每个请求在协议层面相对独立。
如果应用需要保持登录状态或用户身份，通常借助 Cookie、Session、Token 等机制。
```

### 6.3 HTTPS

HTTPS 可以理解为：

```text
HTTPS = HTTP + TLS/SSL
```

提供：

```text
加密
身份认证
完整性保护
```

### 6.4 HTTP 和 HTTPS 区别模板

```text
HTTP 是明文传输协议，默认端口 80，数据在传输过程中容易被窃听或篡改。
HTTPS 在 HTTP 基础上加入 TLS/SSL，默认端口 443，
提供加密传输、服务器身份认证和数据完整性保护，因此安全性更高。
```

### 6.5 常见陷阱

| 题干 | 判断 |
|---|---|
| HTTPS 不需要 HTTP | 错，HTTPS 是 HTTP 加安全层 |
| HTTP 是有状态协议 | 错，HTTP 本身无状态 |
| Cookie 可帮助 HTTP 维持状态 | 对 |
| HTTPS 默认端口 80 | 错，HTTPS 默认 443 |

### 6.6 HTTP 状态码（选择/判断常出）

| 状态码 | 含义 | 说明 |
|:-----:|------|------|
| **200** | OK | 请求成功 |
| **301** | 永久重定向 | 资源换了新地址，以后用新地址 |
| **302** | 临时重定向 | 资源临时在别处，下次还用原地址 |
| **304** | Not Modified | 缓存未过期，直接用缓存（结合缓存机制） |
| **400** | Bad Request | 请求语法错误，服务器看不懂 |
| **403** | Forbidden | 服务器拒绝执行 |
| **404** | Not Found | 资源不存在 |
| **500** | Internal Server Error | 服务器内部出错 |
| **502** | Bad Gateway | 代理/网关收到上游无效响应 |
| **503** | Service Unavailable | 服务器暂时无法处理（过载/维护） |

> **速记口诀：** 2xx成功 3xx重定向 4xx客户端错 5xx服务端错

### 6.7 GET vs POST 区别（简答常考）

| 对比维度 | GET | POST |
|---------|-----|------|
| 参数位置 | URL 中（?key=value） | 请求体（Body）中 |
| 可见性 | 明文可见，会留在浏览器历史 | 不可见 |
| 长度限制 | 有（URL 长度限制，通常 2KB） | 无 |
| 安全性 | 低（参数暴露在 URL） | 相对较高 |
| 幂等性 | **幂等**（多次请求结果相同） | **不幂等**（多次请求可能不同） |
| 可缓存 | ✅ 浏览器可缓存 | ❌ 不缓存 |
| 可书签 | ✅ 可收藏为书签 | ❌ 不可以 |
| 数据类型 | 仅 ASCII | 任意类型（二进制也可） |

> **简答题模板：**
> ```text
> GET 和 POST 是 HTTP 最常用的两种请求方法。
> GET 参数放在 URL 中，有长度限制，安全性较低但可被缓存和收藏为书签，且是幂等的。
> POST 参数放在请求体中，无长度限制，安全性较高，但不被缓存，且不幂等。
> GET 通常用于获取资源，POST 通常用于提交数据（如表单、上传）。
> ```

### 6.8 HTTP 版本进化 + 持久连接

```text
HTTP/0.9 (1991)：只有 GET，纯文本
HTTP/1.0 (1996)：增加 POST、状态码、Header，每个请求一个TCP连接
                   ↓ 问题：一个页面有10张图片就要建10次TCP连接
HTTP/1.1 (1997)：默认持久连接（Keep-Alive），支持管道化
                   ↓ 优化：同一个TCP可以发多个请求，但队头阻塞仍存在
HTTP/2   (2015)：多路复用、头部压缩、服务器推送
                   ↓ 解决：一个TCP并发传多个流，不再队头阻塞
HTTP/3   (2022)：基于 QUIC（UDP），解决 TCP 级别的队头阻塞
```

**核心对比（考试重点）：**

| | 非持久连接（HTTP/1.0） | 持久连接（HTTP/1.1 默认） |
|--|----------------------|------------------------|
| 行为 | 每个对象建立一次TCP | 同一TCP传多个对象 |
| 效率 | 低（多次握手+慢启动） | 高（复用连接） |
| 默认 | HTTP/1.0 默认非持久 | HTTP/1.1 默认持久 |

> **考试答题模板：**
> ```text
> 持久连接（Keep-Alive）让多个 HTTP 请求复用同一个 TCP 连接，
> 减少了重复建立连接带来的三次握手和慢启动开销，
> 从而降低了访问延迟，提高了传输效率。
> HTTP/1.1 默认使用持久连接，而 HTTP/1.0 默认使用非持久连接。
> ```

**速记口诀：**

```text
HTTP 版本进化：0.9 → 1.0（每图一连接）→ 1.1（持久连）→ 2（多路复用）→ 3（QUIC）
每个版本解决什么问题：慢 → TCP次数太多 → 队头阻塞 → TCP也队头阻塞
```

## 7. FTP 题型

### 7.1 FTP 的双连接

FTP 使用两个连接：

```text
控制连接：传输命令和响应，常用 TCP 21。
数据连接：传输文件数据，按需建立。
```

### 7.2 为什么使用双连接

**真实原因：带外（Out-of-band）控制 —— 控制与数据分离，互不阻塞。**

```text
如果没有双连接，FTP 会这样：
  传输大文件时（比如 1GB 电影）→ 数据一直在 TCP 流里发
  → 你想"取消传输"或"切目录" → 控制命令也在同一条流里
  → 必须等数据发完，命令才能被处理

双连接就解决了这个问题：
  控制连接（21）→ 传命令，随时可用
  数据连接（20/随机）→ 传文件，用完即关
  
  传文件的时候，你照样可以发"CANCEL"、"CD .."等命令
  因为控制命令走的是另一条独立的 TCP 连接！
```

**类比理解：**
```text
单连接 ≈ 只有一条马路，货车（数据）和小轿车（命令）一起堵着
双连接 ≈ 专门给货车修一条路，小轿车走另一条路，互不干扰
```

**考试答题模板（直接背）：**
```text
FTP 采用双连接设计，核心原因是为了实现带外控制。
控制连接（TCP 21）专门传输命令，在整个会话期间保持打开；
数据连接（TCP 20 或随机端口）按需建立，传完即关。
这样在传输大文件时，用户仍可发送控制命令，
控制与数据互不阻塞，保证控制会话的实时性。
```

### 7.3 主动/被动模式基线理解

主动模式：

```text
客户端告诉服务器自己的数据端口，服务器主动连接客户端的数据端口。
```

被动模式：

```text
服务器开放一个数据端口，客户端主动连接服务器的数据端口。
```

考试不一定深考，但看到 `PORT/PASV` 要知道和 FTP 数据连接建立方式有关。

## 8. 电子邮件协议题型

### 8.1 三个协议

| 协议 | 作用 | 关键词 |
|---|---|---|
| SMTP | 发送邮件 | 客户端到邮件服务器、邮件服务器之间转发 |
| POP3 | 接收邮件 | 下载到本地，较简单 |
| IMAP | 接收/管理邮件 | 在线管理，多设备同步更方便 |

### 8.2 必背句

```text
SMTP 负责发邮件。
POP3 和 IMAP 负责收邮件。
IMAP 更适合在服务器上在线管理邮件，多设备同步体验更好。
```

### 8.3 MIME 和 base64

为什么需要 MIME：

```text
传统邮件主要支持简单文本，MIME 扩展了邮件格式，使邮件可以携带非 ASCII 文本、图片、音频、附件等多种内容。
```

base64：

```text
把二进制数据转换成文本形式，便于在只适合传输文本的系统中传输。
3 字节通常编码成 4 个字符，体积约增加 33%。
```

### 8.4 常见陷阱

| 题干 | 判断 |
|---|---|
| SMTP 主要用于接收邮件 | 错 |
| POP3 和 IMAP 都与接收邮件有关 | 对 |
| MIME 用于扩展邮件可承载内容类型 | 对 |
| base64 编码会压缩数据 | 错，通常会膨胀 |

## 9. Cookie、缓存、CDN 题型

### 9.1 Cookie

```text
Cookie 是服务器通过浏览器保存在客户端的小段信息，
常用于用户识别、登录状态维持、个性化设置等。
```

### 9.2 Web 缓存

```text
Web 缓存保存用户访问过的资源副本。
再次访问相同资源时，可直接从缓存获取，减少访问时延，降低服务器和网络负载。
```

### 9.3 CDN

```text
CDN 是内容分发网络，通过在不同地理位置部署缓存节点，
让用户从距离更近的节点获取资源，提高访问速度并减轻源站压力。
```

### 9.4 对比表

| 机制 | 解决什么问题 |
|---|---|
| Cookie | HTTP 无状态下的状态保持 |
| Web 缓存 | 重复资源访问慢、服务器压力大 |
| CDN | 用户离源服务器远、访问延迟高 |

### 9.5 提高访问速度/降低延迟（高频题型）

**考题不会直接问"缓存是什么"，而是问：**

| 考题实际问法（伪装版） | 真实考点 |
|----------------------|---------|
| "视频网站如何让全国用户都流畅观看？" | CDN 就近分发 |
| "再次访问同一个网页为什么比第一次快？" | Web 缓存 + 持久连接 |
| "某网站用户量暴增，服务器压力大，怎么优化？" | Web 缓存 / CDN |
| "如何减少用户访问资源的响应时间？" | 缓存 + CDN + 持久连接 |
| "多个用户访问同样资源，怎样避免重复请求源服务器？" | Web 缓存 / CDN |

**应答核心逻辑（三步递进）：**

```text
第一步：Web 缓存（浏览器/代理缓存）
  "把用户访问过的资源存在本地或代理服务器上，
  下次访问相同资源直接取缓存，不去源服务器请求。
  减少重复请求 → 降低时延 + 减轻源站压力。"

第二步：CDN（内容分发网络）
  "如果用户和源服务器距离很远（比如新疆访问北京的服务器），
  CDN 在全国各地部署缓存节点，
  让用户从最近的节点获取资源。
  缩短物理距离 → 降低传播时延。"

第三步：持久连接（HTTP/1.1 Keep-Alive）
  "在同一个 TCP 连接上传输多个 HTTP 请求响应，
  避免重复建立连接的开销。
  减少三次握手 → 降低连接建立时延。"
```

**考试综合答题模板：**

```text
提高网络访问速度、降低延迟的手段主要包括：

1. Web 缓存：缓存用户已访问的资源副本，再次访问时直接命中缓存，
   减少重复请求和网络传输。

2. CDN：将内容分发到离用户更近的边缘节点，用户就近获取资源，
   缩短传播距离，降低响应时间。同时分担源服务器压力。

3. 持久连接（HTTP Keep-Alive）：让多个 HTTP 请求复用同一个 TCP 连接，
   减少三次握手和慢启动带来的额外时延。

实际中通常三者结合使用：CDN 做全局加速，Web 缓存做局部加速，
持久连接优化单次会话效率。
```

这类题经常是综合题，可以按层次写。

标准流程：

```text
1. 用户在浏览器输入 URL。
2. 浏览器先通过 DNS 查询域名对应的 IP 地址。
3. 获得 IP 后，浏览器与服务器建立 TCP 连接。
4. 如果是 HTTPS，还要进行 TLS 握手，协商加密参数并验证证书。
5. 浏览器发送 HTTP/HTTPS 请求。
6. 服务器返回响应报文和网页资源。
7. 浏览器解析 HTML，并继续请求 CSS、JS、图片等资源。
8. 页面渲染完成。
```

## 12. 简答模板库

### DNS 的作用是什么

```text
DNS 用于将人容易记忆的域名解析为网络通信所需的 IP 地址。
客户端通常先向本地域名服务器发起查询；
若本地域名服务器没有缓存结果，则继续通过递归或迭代方式查询根域名服务器、顶级域名服务器和权威域名服务器，
最终获得目标主机 IP 地址。
```

### HTTP 和 HTTPS 区别

```text
HTTP 是明文传输协议，默认端口 80。
HTTPS 是在 HTTP 基础上加入 TLS/SSL，提供加密、身份认证和完整性保护，默认端口 443。
因此 HTTPS 安全性更高，但建立连接和加密处理开销也更大。
```

### FTP 为什么使用双连接

```text
FTP 使用控制连接传输命令和响应，使用数据连接传输文件内容。
控制连接通常使用 TCP 21 端口并在会话期间保持；
数据连接按需建立，用于实际文件传输。
这种设计使控制信息和文件数据相互独立。
```

### HTTP 为什么无状态

```text
HTTP 协议本身不会自动保存客户端前后请求的状态，每个请求在协议层面相对独立。
为了实现登录保持、购物车等功能，Web 应用通常借助 Cookie、Session 或 Token 维护状态。
```

### Cookie、缓存、CDN 的作用

```text
Cookie 用于在客户端保存状态信息，帮助 HTTP 应用识别用户。
Web 缓存用于保存资源副本，减少重复请求和访问时延。
CDN 通过分布式节点就近提供内容，提高访问速度并减轻源站压力。
```

### P2P 和 C/S 架构的区别

```text
C/S 架构中，客户端向中央服务器请求资源，服务器压力随用户数增加而增大，可扩展性差。
P2P 架构中，每个节点既是客户端也是服务器，下载的同时也上传，
因此节点越多、系统总资源越丰富，具有自扩展能力。
P2P 适合文件分发场景（如 BitTorrent），但安全性和可控性不如 C/S。

P2P 的三种实现方式：
- 集中式：有中心索引服务器（Napster）
- 完全分布式：无中心、全靠邻居转发（Gnutella）
- 混合式：有 Tracker 但传输是 P2P（BitTorrent）
```
