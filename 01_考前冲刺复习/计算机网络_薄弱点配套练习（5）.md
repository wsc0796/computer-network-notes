# 1. 编码识图（配套 §1.2）

## 题1-1（识图·基础）

下图是某段位序列的编码波形，请判断属于哪种编码方式：

```text
位序列：  1    0    1    1    0    0    1

波形A：
     ┌──┐     ┌─┐  ┌─┐        ┌──┐
     │  │     │ │  │ │        │  │
─────┘  └─────┘ └──┘ └────────┘  └────
（特征：连续0处是水平直线，每个位周期电平不变）

波形B：
  ┌┐    ┌──┐ ┌┐               ┌┐
  ││    │  │ ││               ││
──┘└────┘  └─┘└───────────────┘└──
  ↑     ↑    ↑                 ↑
 跳变=1 不变=0 跳变=1         跳变=1
（特征：位边界有跳变=1，无跳变=0）

波形C（曼彻斯特）：
  ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐ ┌┐
  ││ ││ ││ ││ ││ ││ ││
──┘└─┘└─┘└─┘└─┘└─┘└─┘└─┘└──
   ↓   ↑   ↓   ↓   ↑   ↑   ↓     ← 每个位中间的跳变方向
   1   0   1   1   0   0   1       （下降沿=1，上升沿=0）
（特征：每个位中间必跳变，跳变方向决定0/1）

波形D（差分曼彻斯特）：
  ┌┐  ┌┐  ┌┐  ┌┐  ┌┐  ┌┐  ┌┐
  ││  ││  ││  ││  ││  ││  ││
──┘└──┘└──┘└──┘└──┘└──┘└──┘└──
  │    ↑   │    │    ↑   ↑   │    ← 位边界：看相邻两位之间的电平
 无跳  有跳 无跳 无跳 有跳 有跳 无跳   有没有翻转
  =1   =0   =1   =1   =0   =0   =1
（特征：每个位中间必跳变 + 位边界跳变=0/无跳变=1）
```

关键区分方法——两步走：
```text
第1步：每个位中间有跳变吗？
  没有 → NRZ 或 NRZ-I（跳到第2a步）
  有   → 曼彻斯特 或 差分曼彻斯特（跳到第2b步）

第2a步（NRZ家族）：看位边界
  NRZ：每个位周期电平不变，连续相同位处是直线
  NRZ-I：位边界处有跳变=1，无跳变=0

第2b步（曼彻斯特家族）：看位边界的跳变
  曼彻斯特：只看位中间的跳变方向（高→低=1，低→高=0），
           位边界不关心（跳不跳都无所谓，只由相邻位的中间跳变自然决定）
  差分曼彻斯特：除了中间必有跳变，还必须看位边界——
           位边界有跳变=0，位边界无跳变=1
```

问题：
(1) 波形A是哪种编码？判据是什么？
(2) 波形B是哪种编码？判据是什么？
(3) 波形C是哪种编码？波形D是哪种编码？两者的核心区别在哪里？

---

**答案与解析：**

(1) 波形A = **NRZ（非归零编码）**

判据：高电平=1，低电平=0。连续相同位（如两个0）处波形是水平直线，没有任何跳变。整个位周期保持电平不变。

(2) 波形B = **NRZ-I（见1就翻）**

判据：在位边界处找跳变。第1位是1→有跳变，第2位是0→无跳变，第3位是1→有跳变（可以看到位边界处的翻转）。注意：NRZ-I只看位边界有没有跳变，中间不管。

(3) 波形C = **曼彻斯特编码**，波形D = **差分曼彻斯特编码**

两者相同点：每个位中间都必有一次跳变。

两者核心区别：
  曼彻斯特 → 看位中间的跳变方向（高→低=1，低→高=0）。
             位边界的跳变不重要，它只是相邻两位中间跳变方向的自然结果。
  
  差分曼彻斯特 → 看位边界的跳变（有跳变=0，无跳变=1）。
                 位中间的跳变只是"每次必翻"的同步信号，不携带数据信息。
                 数据信息在位边界处！

考试速判口诀：
  "中间跳变看方向 = 曼彻斯特"
  "中间跳变看边界 = 差分曼彻斯特"

## 题1-2（识图·进阶）

已知位序列 `0 1 0 0 1 1`，以下哪个编码在长连续0时会出现同步问题？

A. NRZ
B. 曼彻斯特编码
C. 差分曼彻斯特
D. NRZ-I

---

**答案：A、D**

NRZ：连续0时波形是一条低电平水平线，接收方无法从波形中提取时钟信号，不知道到底有几个0。

NRZ-I：虽然遇到1会翻转，但连续0时电平保持不变，同样没有跳变，同步会丢失。

曼彻斯特和差分曼彻斯特：**每个位中间都必有一次跳变**，所以无论位序列是什么，始终能从跳变中恢复时钟。这是它们最大的优点。

---

## 题1-3（编码·判断）

判断题：曼彻斯特编码的波特率等于比特率。（ ）

---

**答案：错**

曼彻斯特编码每个位周期中间有一次跳变，所以一个比特对应两个码元（一个高电平+一个低电平，或反之）。因此 **波特率 = 2 × 比特率**。

这就是曼彻斯特的代价：自带时钟同步，但带宽利用率只有50%。

# 2. 奈奎斯特与香农（配套 §1.4）

## 题2-1（奈奎斯特·基础）

一条无噪声信道带宽为 8kHz，采用四进制编码（V=4）。该信道的最大数据传输速率是多少？

---

**答案：**

```text
C = 2W × log₂(V)
  = 2 × 8000 × log₂(4)
  = 16000 × 2
  = 32000 bps = 32 kbps
```

---

## 题2-2（香农·基础）

一条信道带宽为 3kHz，信噪比为 30dB。求该信道的最大数据传输速率（取整到 kbps）。

---

**答案：**

```text
第1步：dB转比值
30 = 10 × log₁₀(S/N)
S/N = 10³ = 1000

第2步：香农公式
C = W × log₂(1 + S/N)
  = 3000 × log₂(1001)
  ≈ 3000 × 10      (log₂(1024)=10, 1001≈1024)
  = 30000 bps = 30 kbps
```

注意：考试时 log₂(1001) ≈ 10 就够了。如果严格算，log₂(1001) ≈ 9.97，结果约 29.9 kbps，取 30 kbps 即可。

---

## 题2-3（奈奎斯特 vs 香农·综合）

某信道带宽为 4kHz，信噪比为 30dB，采用 256 种电平编码。

(1) 按奈奎斯特公式，最大数据率是多少？
(2) 按香农公式，最大数据率是多少？
(3) 该信道的实际最大数据率是多少？

---

**答案：**

```text
(1) 奈奎斯特：
    C = 2 × 4000 × log₂(256) = 8000 × 8 = 64000 bps = 64 kbps

(2) 香农：
    S/N = 1000 (30dB)
    C = 4000 × log₂(1001) ≈ 4000 × 10 = 40000 bps = 40 kbps

(3) 实际最大 = min(64k, 40k) = 40 kbps

瓶颈是噪声，不是电平数。即使增加到1024种电平，数据率也超不过40 kbps。
这就是香农极限的物理意义——噪声决定了信道的理论上限。
```

---

## 题2-4（波特率 vs 比特率）

某调制解调器采用 16 种不同的码元，波特率为 2400 Baud。比特率是多少？如果改用 64 种码元，比特率变为多少？

---

**答案：**

```text
16种码元：比特率 = 2400 × log₂(16) = 2400 × 4 = 9600 bps
64种码元：比特率 = 2400 × log₂(64) = 2400 × 6 = 14400 bps

注意：波特率保持不变（每秒还是2400个码元），
但每个码元携带更多比特，所以比特率提高了。
代价是：64种码元 → 每种码元的电平差异更小 → 更容易受噪声干扰。

这也是奈奎斯特和香农之间的张力：
奈奎斯特说"多用电平可以提速"，
香农说"噪声决定了你最多能用多少种电平"。
```

# 3. GBN 与 SR（配套 §2）

## 题3-1（GBN·窗口判断）

采用 GBN 协议，帧序号用 3 位表示（0~7循环）。发送窗口大小设为 4。

某时刻发送方已发送帧 3,4,5,6 但尚未收到确认，此时收到 ACK=4（对帧3的累积确认）。

(1) 发送窗口如何滑动？
(2) 发送方接下来可以发送哪些序号的帧？

---

**答案：**

```text
(1) ACK=4 表示接收方已正确收到帧0,1,2,3。（累计确认）
    发送窗口左边界滑到4。
    发送窗口右边界滑到 4+4-1 = 7。
    发送窗口现在覆盖：4,5,6,7

(2) 帧4,5,6已经发出但未确认（仍在窗口内）。
    帧7在窗口内但尚未发送 → 可以发送帧7。

窗口状态变化图示：
  之前：[3,4,5,6]  （窗口覆盖第3~6帧）
  收到 ACK=4 后：[4,5,6,7]  （窗口右移一位）
```

---

## 题3-2（GBN·重传场景）

GBN协议，发送窗口=4，发送方已发送帧 0,1,2,3,4,5,6（分两轮窗口）。

场景：
- 接收方正确收到帧0,1 → 发 ACK=1, ACK=2
- 帧2丢失
- 接收方收到帧3 → 帧3不是期望的帧2 → **丢弃帧3并重发 ACK=2**
- 接收方收到帧4 → 同上，丢弃，重发 ACK=2
- 发送方收到多个 ACK=2

问题：
(1) 发送方最终会重传哪些帧？
(2) 如果帧2丢失前，窗口内还有帧2,3,4,5已发出，超时后会重传哪些帧？
(3) 这样"白传"的帧有几个？

---

**答案：**

```text
(1) 发送方会重传从帧2开始的所有已发未确认帧。

    假设计时器超时时，窗口是 [2,3,4,5]（含已发出的）。
    重传帧：2,3,4,5（从丢失帧开始，全部重发）

(2) 同(1)，重传帧2,3,4,5。

(3) 帧3,4,5都被接收方丢弃了 → 3个帧白传了。
    帧2的一个丢包，导致了3个正确帧的浪费。

这就是 GBN 最大的问题：一个包丢了，后面正确的全白传。
这也是为什么 SR 被发明出来的原因。
```

---

## 题3-3（SR·窗口判断）

采用 SR 协议，帧序号用 3 位表示。发送窗口=4，接收窗口=4。

接收方已缓存了乱序到达的帧 1,3,4，正在等待帧0。

(1) 帧0到达后，接收方如何操作？
(2) 接收窗口如何滑动？
(3) SR 协议下接收窗口最大是多少？

---

**答案：**

```text
(1) 帧0到达 → 接收方验证帧0正确 → 缓存中已有 [0,1,3,4]
    因为0,1连续 → 将帧0和帧1按序交付给上层
    帧3,4尚不连续（缺帧2），继续缓存
    接收窗口左边界滑到2

(2) 交付0,1后，接收窗口从 [0,1,2,3] 滑到 [2,3,4,5]

(3) SR 协议接收窗口最大 = 2^(n-1) = 2^(3-1) = 4

为什么不是7？
如果接收窗口=7，就会出现新旧两轮的帧序号重叠，
接收方无法区分"收到的是重传的旧帧0"还是"新一轮的新帧0"。
减半就是为了避免这个"序号回绕"的歧义。
```

---

## 题3-4（GBN vs SR·简答）

简述：为什么 GBN 只需要一个计时器，而 SR 需要为每个帧设置独立的计时器？

---

**答案：**

```text
GBN 用一个计时器管整个发送窗口：
  超时 → 从窗口最左边（最早未确认的帧）开始全重传。
  因为 GBN 的重传是"回退N帧"，一个计时器就够了。

SR 必须每个帧独立计时：
  因为 SR 只重传真正丢失的那个帧（不是全部重传）。
  如果只用一个计时器，超时时不知道该重传哪个。
  所以要每个帧各自计时，哪个帧超时就重传哪个。

本质区别：
  GBN 超时 = "窗口里最早的帧没确认，全来一遍"
  SR 超时  = "这个具体的帧没确认，就重传这个"
```

# 5. UDP 首部解析（配套 §4）

## 题4-1（UDP·十六进制解析）

捕获到 UDP 报文首部十六进制如下：

```text
C0A8 0050 0030 B4F2
```

(1) 源端口和目的端口分别是多少（十进制）？
(2) UDP 总长度多少字节？数据部分多少字节？
(3) 目的端口 0050 对应的是什么服务？

---

**答案：**

```text
(1) 源端口 C0A8₁₆：
    C×4096 + 0×256 + A×16 + 8
    = 12×4096 + 0 + 10×16 + 8
    = 49152 + 0 + 160 + 8
    = 49320

    目的端口 0050₁₆：
    = 5×16 + 0 = 80

(2) 总长度 0030₁₆：
    = 3×16 + 0 = 48 字节

    数据长度 = 总长度 - 首部(8字节) = 48 - 8 = 40 字节

(3) 端口 80 是 HTTP 的知名端口号。
    
    但要注意：传统 HTTP 基于 TCP，不是 UDP。
    本题捕获到的是 UDP 报文且目的端口=80，可能是：
    ① HTTP/3（QUIC）使用 UDP 端口 443，不是 80
    ② 某些实验性或非标准服务使用 UDP 80
    ③ 更常见的 UDP 知名端口：53(DNS)、67/68(DHCP)、69(TFTP)、161(SNMP)
    
    考试如果问"端口80对应什么服务"→ 答"HTTP的知名端口号"。
    如果是综合题问"这个UDP报文最可能是什么"→ 别直接写 HTTP请求/响应，
    因为 HTTP/1.1 和 HTTP/2 都跑在 TCP 上。
```

速算诀窍：
```text
C000₁₆ = 49152
00A8₁₆ = 168
C0A8₁₆ = 49152 + 168 = 49320

或者直接用：C0A8₁₆ = 0xC0A8 = 49320
如果考试允许用计算器这更快；不允许就用逐位展开。
```

---

## 题4-2（UDP·判断服务类型）

以下 UDP 报文首部的目的端口字段解析结果，判断各对应什么服务：

(1) 0035
(2) 0043
(3) 0069
(4) 01BB

---

**答案：**

```text
(1) 0035₁₆ = 53 → DNS
(2) 0043₁₆ = 67 → DHCP（服务器端口，客户端是68=0044₁₆）
    注意：DHCP 使用 UDP 67(服务器) 和 68(客户端)
(3) 0069₁₆ = 105 → 不是 TFTP！TFTP 是 69₁₀ = 0045₁₆
    0069₁₆ = 6×16+9 = 105，通常不分配给固定知名服务
    陷阱：69₁₀ ≠ 0069₁₆！
(4) 01BB₁₆ = 1×256 + 11×16 + 11 = 256 + 176 + 11 = 443 → HTTPS

高频端口速查（十六进制 → 十进制 → 服务 → 传输层协议）：
  0015₁₆=21 → FTP控制(TCP)
  0014₁₆=20 → FTP数据(TCP)
  0019₁₆=25 → SMTP(TCP)
  0035₁₆=53 → DNS(UDP为主，大查询用TCP)
  0044₁₆=68 → DHCP客户端(UDP)
  0045₁₆=69 → TFTP(UDP)
  0050₁₆=80 → HTTP(TCP)
  006E₁₆=110 → POP3(TCP)
  008F₁₆=143 → IMAP(TCP)
  01BB₁₆=443 → HTTPS(TCP)
```

## 题4-3（UDP·综合判断）

某 UDP 报文首部：`1F90 0035 002C 8A13`

(1) 这个 UDP 报文最可能是什么用途？（根据端口判断）
(2) 源端口可能是临时端口还是知名端口？
(3) UDP 数据部分长度是多少？

---

**答案：**

```text
(1) 解题最快思路：看目的端口。
    0035₁₆ = 53 → DNS 的知名端口！
    所以这极可能是 DNS 查询报文。

(2) 源端口 1F90₁₆ = 1×4096 + 15×256 + 9×16 + 0
    = 4096 + 3840 + 144 = 8080
    8080 落在 49152~65535 范围之外...
    
    等等，重新算：1F90₁₆
    = 1×16³ + 15×16² + 9×16 + 0
    = 1×4096 + 15×256 + 144
    = 4096 + 3840 + 144
    = 8080

    8080 是注册端口范围内（1024~49151），也可能是临时分配。
    对于 DNS 查询，客户端通常使用临时端口（>1023），
    8080 恰好是 HTTP 备用端口，但在此场景下是客户端临时端口。

    → 源端口 8080 是客户端临时端口。

(3) 总长度 002C₁₆ = 2×16 + 12 = 44 字节
    数据部分 = 44 - 8 = 36 字节
```

# 6. base64编码（配套 §3.4）

## 题5-1（base64·计算）

(1) 3个原始字节经过base64编码后变成几个字符？
(2) 1KB原始数据经过base64编码后体积约为多少？
(3) 1个原始字节（8bit）经过base64编码后变成几个字符？是否需要填充？

---

**答案：**

```text
(1) 3 字节(24bit) → 切成 4 个 6bit 组 → 4 个 base64 字符。

(2) 1KB ≈ 1024 字节
    每 3 字节 → 4 字符
    1024 / 3 ≈ 341.3，即有 341 组完整 + 1 个剩余字节
    341 × 4 = 1364 字符
    剩余 1 字节 → 2 个 base64 字符 + 1 个填充符 "="→ 2 字符
    但这里是 1KB=1024字节：1024=341×3+1，剩1字节变成2字符+2个"="
    
    粗略估算：体积膨胀 = 4/3 ≈ 1.33 倍
    1KB → 约 1.33KB

(3) 1个原始字节(8bit)：
    切成 1 个 6bit 组 + 剩余 2bit（不够6bit，补4个0）
    → 共 2 个 6bit 组 → 2 个 base64 字符
    → 补 2 个 "=" 填充到 4 的倍数 → 共 4 字符输出
    
    所以 1 字节 → 4 个字符 + 2 个 "=" → 输出 "XX==" 的形式。
    这就是 base64 最差情况：体积膨胀约 4 倍。
```

## 题5-2（MIME·简答）

简述 MIME 的作用以及为什么 SMTP 需要 MIME。

---

**答案：**

```text
SMTP 最初只能传输 7 位 ASCII 文本，不能传输：
  - 非英文文字（如中文、日文）
  - 二进制文件（图片、PDF、音频）
  - 超过 1000 字符的长行

MIME 扩展了邮件格式：
  1. 增加 Content-Type 头，标识内容类型（text/html, image/jpeg等）
  2. 增加 Content-Transfer-Encoding，指定编码方式（最常用 base64）
  3. 支持 multipart，一封邮件包含多个部分（正文+附件）

一句话：MIME 让电子邮件能携带多媒体和二进制内容。
```

# 8. IPv6（配套 §5）

## 题7-1（IPv6·简答）

IPv6 为什么取消了路由器分片？如果 IPv6 数据报大于路径 MTU 怎么办？

---

**答案：**

```text
IPv4 路由器分片的问题：
  1. 分片和重组消耗路由器 CPU
  2. 一片丢失 → 整个数据报作废 → 其他片白传
  3. 分片使路由器处理变慢，难以达到线速转发

IPv6 的设计：
  - 路由器不再分片
  - 如果数据报 > 出链路 MTU → 路由器丢弃它
  - 并向源主机发送 ICMPv6 "Packet Too Big" 报文
  - 源主机收到后，用更小的 MTU 重新发送
  - 源主机通过 Path MTU Discovery 提前探测路径最小 MTU

本质：把分片的负担从"网络核心"推到"网络边缘"（源主机）。
这符合 Internet 端到端的设计原则——复杂的逻辑放在终端，路由器保持简单。
```

---

## 题7-2（IPv6·地址简写）

将以下 IPv6 地址简写：

```text
2001:0db8:0000:0000:0000:ff00:0042:8329
```

---

**答案：**

```text
步骤1：去掉前导零
2001:db8:0:0:0:ff00:42:8329

步骤2：连续的零块用 :: 替代（只能用一次）
      中间有三个连续的 0
2001:db8::ff00:42:8329

注意：
- 前导零可以省略：0db8 → db8, 0042 → 42
- 连续的零用 ::（如果有多段连续零，:: 只能替换最长的那段）
- :: 只能用一次，否则地址会有歧义
```

# 9. 综合自测（10分钟）

以下 5 题限时 10 分钟闭卷完成，然后对答案。

## 综合题1

数字信号编码中，哪种编码方式在每个位周期中间一定有一次跳变？
A. NRZ  B. NRZ-I  C. 曼彻斯特  D. 以上都有

## 综合题2

一条带宽 2kHz、信噪比 20dB 的信道，按香农公式的最大数据率约为多少 kbps？

## 综合题3

采用 GBN 协议，3 位序号，发送窗口大小 = 7。已发送帧 0~6。帧 2 丢失。
接收方收到帧 3 时会怎么做？

## 综合题4

UDP 报文首部：`1A0B 0050 0028 7F3C`，目的端口是多少？该报文可能是什么服务？

## 综合题5

IPv6 中，路由器收到一个大于出链路 MTU 的数据报时：
A. 对数据报分片后转发
B. 丢弃该数据报并发送 ICMPv6 差错报文
C. 缓存数据报等待 MTU 增大
D. 通过 IP 选项扩展数据报

---

**答案：**

```text
1. C（曼彻斯特和差分曼彻斯特每个位中间都有跳变，NRZ/NRZ-I没有）

2. 20dB → S/N=100
   C = 2000 × log₂(101) ≈ 2000 × 6.66 ≈ 13.3 kbps
   近似值：2000 × log₂(128) = 2000 × 7 = 14 kbps

3. 接收方期望帧2，收到帧3（不是2）→ 丢弃帧3，重发 ACK=2

4. 目的端口 0050₁₆ = 80（HTTP的知名端口号）
   数据部分 = 0028₁₆ - 8 = 40 - 8 = 32 字节
   注意识别：端口号80对应HTTP的知名端口，但传统HTTP基于TCP而不是UDP。
   如果是UDP报文且目的端口=80，不是典型的HTTP请求/响应。
   此题考点是"端口号→服务识别"而非"UDP承载的应用"。

5. B
   IPv6 路由器不进行分片。若数据报大于出链路 MTU，则丢弃并向源主机
   发送 ICMPv6 "Packet Too Big" 报文，由源主机重新以更小 MTU 发送。
```
